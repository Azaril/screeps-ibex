# Competitive Analysis — Ibex vs Overmind (design & strategy pass)

- **Type:** Analysis note (NOT an ADR — decision-bearing calls are deferred to the ADRs it cross-references).
- **Date:** 2026-06-09
- **Related:** review report `../reviews/ibex-review-report.md` (IBEX-001..050, §8 rewrite direction), rewrite plan `../plans/rewrite-plan.md`, ADRs [0001](0001-entity-model.md)–[0006](0006-eval-and-iteration-harness.md), prior-art notes in `../references/`.
- **Subject:** Overmind (Ben Bartlett, MIT-licensed TypeScript). **Inspiration only — capture WHAT/WHY, do not copy code.** Combat specifics are source-derived (the author keeps combat deliberately vague); economy/logistics specifics are blog/wiki-cited.

> **Purpose.** A second pass on overall design and strategy: where does Ibex sit relative to the most-studied open-source Screeps bot, and which of Overmind's ideas are worth adopting (vs. avoiding, vs. not-applicable) given Ibex's *actual* code and the rewrite already planned? This note does **not** introduce new decisions; it maps gaps onto the existing findings/ADRs and flags the few that are genuinely NEW design work. Per the review, Ibex is a competently-designed, feature-complete economic bot that is **not yet top-tier** and is **strategically non-viable at competitive scale until the CPU death-spiral is closed** (review §Exec). That survival verdict bounds everything below: a logistics or combat upgrade is worthless on a colony that dies to its own CPU.

## 1. Architecture mapping

Overmind runs a global **Build → Init → Run** loop: Build caches + instantiates the hierarchy, Init gathers prioritized *requests* (spawn/transport/link), Run executes state changes. The **Overseer** is a single priority-queue scheduler ("process table") that runs every directive + overlord by priority and *places new directives reactively* on stimuli. Spawning is **inverted** vs. role bots: the **Overlord (the mission)** owns creeps and submits prioritized spawn *requests*; the Hatchery cluster decides what actually spawns. There is **no static creep role** — creeps are `Zerg` wrappers re-bound to overlords each tick via `recalculateCreeps()`.

| Ibex concept | Overmind analog | Structurally similar / weaker / different |
|---|---|---|
| **operation** (`operations/data.rs`: MiningOutpost, Claim, Colony, Attack, Scout, **War**) | **Directive** (flag-anchored persistent goal) + the colony-wide Overlords it spawns | **Similar in intent, different in trigger.** Ibex operations are ECS singletons managed by `managersystem.rs` (respawns missing singletons); Overmind directives are flag-anchored and the Overseer *places them reactively on detected stimuli* (invasions, expansion targets). Ibex's `WarOperation` is the closest analog to a reactive directive-tree but **returns `Running` forever with no child-abort** (IBEX-002, war.rs:1441) — weaker supervision than the Overseer's place/`remove()` lifecycle. |
| **mission** (`missions/data.rs`: 28 variants, room/objective-scoped) | **Overlord** ("a process": mine a source, guard, bootstrap) | **Similar — this is Ibex's strongest structural match.** Both are the per-objective "process" that owns work end-to-end. **Weaker on two axes:** (a) Ibex missions do **not** themselves own the declarative spawn-reconciliation Overmind centralizes in `wishlist()` (see §2 lifecycle); (b) a single `Err` from any mission state tears the whole mission down (IBEX-042) where an Overlord/directive degrades gracefully. |
| **job** (`jobs/data.rs`: Harvest/Build/Haul/Upgrade/Claim/Reserve/Dismantle/Scout/StaticMine/LinkMine/**SquadCombat**) | **Zerg + Task** (contextual creep wrapper, "do X to Y until Z") — *no static role* | **Different and arguably weaker by design.** Ibex jobs ARE effectively static per-creep roles assigned at spawn (`JobData` is serialized per creep). Overmind deliberately has **no static role**: a creep is re-bound to whatever overlord needs it via `setOverlord()`/`recalculateCreeps()` each tick. Ibex's fixed job-per-creep is simpler but loses the idle-reassignment flexibility (§2). |
| **squad** (`military/squad.rs` `SquadContext`; `military/formation.rs`; jobs `SquadCombatJob`) | **Swarm + SwarmOverlord** (`staticFormation` matrix, atomic `move()`) | **Similar concept, materially weaker execution.** Overmind's `Swarm.move()` is **atomic** (if any member can't move, all moves cancel) keeping formation intact; Ibex's `formation.rs:163` issues **N independent `move_to(slot).range(0)`** toward a soft-gated virtual anchor and the live defense path wires **no formation at all** (`squad_defense.rs:455` `squad_entity=None`) — IBEX-001. This is the single biggest behavioral gap. |
| **transfer queue** (`transfer/transfersystem.rs`: priority/type-masked greedy matching, `select_pickup_and_delivery` per idle hauler) | **LogisticsNetwork** (universal transport role; Gale–Shapley **stable matching** over dq/dt preferences) | **Different and weaker, but bounded.** Ibex models signed supply/withdraw requests with priorities (`TransferPriority` High/Med/Low/None × `TransferType` Haul/Link/Terminal/Use) — conceptually close to Overmind's signed-amount requests. But matching is **greedy per-hauler over small per-hauler room sets** (`pickup_rooms = &[self.room_data]`, haul.rs:289), NOT a room-global stable matcher with predictive coordination. The review **refuted** the worst-case CPU framing (IBEX-030 adjusted High→Medium; IBEX-031 nondeterminism refuted — reservations are accounting-safe), so the gap is *quality of assignment* (no anti-thrash, no predictive load-balancing), not correctness. |
| *(scheduler)* `for_each_system!` fixed-order specs dispatch (`game_loop.rs`) | **Overseer** single priority-queue scheduler | **Different — Ibex has no priority/shed dimension today.** Overmind's single priority queue is exactly the seam ADR [0005](0005-runtime-and-scheduling-model.md) introduces (explicit ordered/priority scheduler) and ADR [0004](0004-cpu-governance-and-load-shedding.md) plugs the `CpuGovernor` into. Ibex's flat ~60-system pass has **no global CPU governor** (IBEX-003) — the extinction-class gap. |
| *(empire resource allocation)* — none | **Strategist** (autonomous colonization scoring) + market auto-buy + `RoadLogistics` | **Absent in Ibex.** No empire-level resource arbiter; expansion eligibility uses **linear (Manhattan) distance**, not route distance (IBEX-032, claim.rs:563 / scout.rs:191). |

**Net:** Ibex's operation→mission→job→squad layering is a *legitimate, recognizable* hierarchy that maps cleanly onto Overmind's Directive→Overlord→Zerg→Swarm — the bones are sound. Ibex is **structurally weaker** in exactly three places the rewrite already targets: (a) no single priority scheduler / CPU governor (the Overseer's defining property), (b) static per-creep jobs vs. declarative mission-owned spawn reconciliation, and (c) the squad/swarm cohesion model.

## 3. Highest-leverage gaps to close (5–8)

Ordered by leverage against the review's objective function (survival-dominating). Each is tied to a finding/ADR or flagged NEW.

1. **G0 — Global CPU governance (PREREQUISITE, dominates all others).** Overmind's defining architectural property is the single priority-queue Overseer with a natural shed point; Ibex has none (IBEX-003/016, extinction-class). **Belongs in: ADR [0004](0004-cpu-governance-and-load-shedding.md) (+[0005](0005-runtime-and-scheduling-model.md) scheduler seam, [0002](0002-serialization.md) seg-55).** Already the rewrite's Increment 1. *No new design — listed first because every other gap below is moot until it lands.*

2. **G1 — Hauling: anti-thrash commitment + predictive load-balancing (NOT full stable-matching).** Adopt Overmind's `predictedRequestAmount` idea (subtract concurrent transporters already targeting a request → implicit load-balancing) and a committed-delivery guard so a hauler finishes its ticket set before re-planning (IBEX-011). **Belongs in: ADR [0003](0003-behavior-modeling.md)** (HaulJob is already the data-driven-FSM pilot; commitment is a state-machine invariant) **+ a transfer-matching note under [0004](0004-cpu-governance-and-load-shedding.md)** for the once-per-tick supply/demand snapshot (IBEX-030). *Extends existing findings; the predictive-amount load-balancing is a NEW behavior worth a short design spike.*

3. **G2 — Squad cohesion primitives from Overmind's Swarm/CombatZerg.** Beyond what [0003](0003-behavior-modeling.md) §B already specifies (wire defense onto `SquadContext`, lead-follower + hard wait-gates, cohesion-as-invariant force-abort), **adopt four concrete Overmind mechanics**: (a) **atomic move** (if any live member can't step, none do) — strengthens the hard wait-gate; (b) **retreat/re-engage hysteresis** (≤~80%→retreat, 100%→re-engage) to stop oscillation; (c) **`isExpired` retire-and-rebuild** when a replacement's lifespan diverges too far from survivors (directly addresses the finite-creep-lifetime constraint); (d) **synchronized `swarmWishlist`** so members spawn in sync and a quad doesn't deploy half-formed. **Belongs in: ADR [0003](0003-behavior-modeling.md) §B** — these specialize its lead-follower decision; (b) and (c) are NEW behaviors to add to the §B migration steps (Increment 4). *Closes Field Report A; (c) also feeds Field Report B teardown.*

4. **G3 — Route-distance expansion gating + incubation.** Route-distance candidate/home-room gating is **IBEX-032** (refactor — reuse the war path's `RoomRouteCache`). **Incubation** (an established colony spawns and feeds a newborn colony's first creeps until it is self-sufficient) is **NEW** — it has no Ibex analog and is high-leverage for converting an economic lead into territory (the review's stated weakness, §Exec). **Belongs in: IBEX-032 (existing); incubation warrants a short new design note or an extension to the Colony/Claim operation design** (not currently in any ADR). Gate behind G0 (route searches must charge the budgeted facade).

5. **G4 — Prespawn-replace-not-renew lifecycle + declarative reconciliation.** Overmind never renews; it prespawns replacements ~(spawnTime+travel+50) ticks before death via `lifetimeFilter`, and `wishlist` reassigns idle creeps before spawning new ones. Ibex renews squad members as cohesion-glue, which **actively prevents** the all-dead terminator (IBEX-002) and is a near-permanent energy drain (engine renew math: ~12 TTL/tick for a 50-part creep). **Belongs in: ADR [0003](0003-behavior-modeling.md)** (stop renewing non-cohesive squads is already there) **+ a NEW reconciler idea**: a mission-level declarative "desired N of body B, prespawn at T-Δ" reconciler. The reconciler is NEW design — it interacts with the static-job model (jobs are assigned at spawn) and should be scoped against ADR [0001](0001-entity-model.md) (creep↔owner binding). *Closes the lifecycle-hang renew-drain.*

6. **G5 — Boost pipeline (wire-or-delete).** `BoostQueue` is inert (IBEX-027); boosted compositions spawn unboosted. **Belongs in: IBEX-027 (existing) + a lab/terminal producer.** If wiring, adopt Overmind's **spawn-time boosting** (boost while the creep is still spawning) to cut congestion and maximize boosted lifetime — relevant to the finite-lifetime constraint. *Strategic capability gap vs boosted opponents; lower survival-leverage than G0–G4, so sequence after cohesion.*

7. **G6 — Market trust hardening + demand-driven auto-buy.** Trailing-window trust model + exposure caps is **IBEX-018** (existing refactor). **Auto-buy missing minerals on a credit threshold** (to complete reaction chains / boosts, feeding G5) is **minor NEW** polish. **Belongs in: IBEX-018 + a small extension to TerminalMission/LabsMission.** Lowest leverage of the set — list it, don't prioritize it.

## 4. Adopt / Avoid / Not-applicable (inspiration, not copying)

**Adopt the idea (already aligned with the rewrite):**
- **Single priority-queue scheduler with a natural shed point** (Overseer) → ADR [0005](0005-runtime-and-scheduling-model.md) scheduler seam + [0004](0004-cpu-governance-and-load-shedding.md) governor. The strongest, most directly-applicable idea.
- **Atomic swarm move, retreat hysteresis, retire-and-rebuild, synchronized squad spawn** → ADR [0003](0003-behavior-modeling.md) §B (G2). Overmind's combat *cohesion mechanics* are exactly the layer Ibex is weakest at, and they are framework-agnostic ideas (not TS code) that drop cleanly onto lead-follower wait-gates.
- **Prespawn-replace-not-renew + idle-creep-reassignment reconciliation** (`wishlist`/`lifetimeFilter`) → G4, complements [0003](0003-behavior-modeling.md)'s "stop renewing non-cohesive squads."
- **Predictive coordination-free load-balancing** (`predictedRequestAmount` subtracts concurrent transporters) → G1, a cheap, single-threaded-friendly anti-thrash idea.
- **Incubation** (colony feeds a newborn) and **route-distance expansion scoring** (Strategist) → G3.

**Avoid / weigh carefully:**
- **Wholesale Gale–Shapley stable matching for hauling.** Overmind itself reports it causes **spiky CPU** when many colonies recompute at once — and Ibex's #1 risk *is* CPU spikes (Field Report C). The review **refuted** the claim that Ibex's greedy matcher is the CPU or correctness problem (IBEX-030 adjusted to Medium; IBEX-031 nondeterminism refuted). Adopting an O(n²) room-global matcher would trade a non-problem for the exact failure mode the rewrite is fighting. **Adopt the predictive *signals*, not the matcher.**
- **Cooperative-async / generator runtime** — Ibex's `screeps-machine` FSM friction (Field Report F) might tempt an async executor, but ADR [0005](0005-runtime-and-scheduling-model.md) already rejects async for a plain priority scheduler; Overmind itself uses a plain priority queue, corroborating that choice. **Do not introduce async.**
- **Flag-based directives.** Overmind anchors directives to colored flags (and hedges against a flag cap). Ibex's ECS-singleton operations are a cleaner fit for a Rust/specs bot; do not retrofit flag-based control. **Not worth adopting.**
- **The Assimilator / cross-bot hivemind** — explicitly out of scope; ignore.

**Not-applicable (Ibex is at parity or the difference is cosmetic):**
- **Economy / mining tree** — Ibex's Colony→LocalSupply tree is mature and reset-resilient (report §2). No change motivated by Overmind.
- **Resumable room planning** — Ibex's seg-60 budgeted planner is already a *strength* Overmind's fixed-layout bunker doesn't have; keep it.
- **StarCraft-themed naming / 5-level hierarchy depth** — purely organizational; Ibex's 4-level operation→mission→job→squad is sufficient.
- **Reactive defense doctrine** — Ibex already builds defense reactively on `has_hostiles`; the difference is bug-level (IBEX-002/033), not doctrine.

## 5. Bottom line

Ibex's architecture is a **legitimate sibling** of Overmind's, not a lesser imitation: the operation→mission→job→squad hierarchy maps cleanly onto Directive→Overlord→Zerg→Swarm, and Ibex's economic brain, tower targeting, and budgeted resumable room planner are **at-or-above parity**. The gaps that matter all cluster in three places the rewrite already targets: (1) **no Overseer-equivalent global scheduler/governor** — the decisive, extinction-class gap ([0004](0004-cpu-governance-and-load-shedding.md)/[0005](0005-runtime-and-scheduling-model.md), G0); (2) **squad/swarm cohesion** — advisory not enforced, defense unwired ([0003](0003-behavior-modeling.md) §B + G2); (3) **lifecycle/spawn doctrine** — renew-as-glue vs. prespawn-replace (G4). The genuinely **NEW** design work Overmind surfaces beyond the existing findings is narrow: **incubation** (G3), a **declarative spawn reconciler** (G4), and the **combat hysteresis / retire-and-rebuild** mechanics (G2) — each small, each gated behind G0. Adopt the *ideas*, not the TypeScript; and specifically **avoid** Overmind's CPU-spiky stable-matching, which would import the one failure mode Ibex must eliminate to be viable at all.
