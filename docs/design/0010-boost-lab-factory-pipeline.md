## Context

The operator asked for deep introspection on **boost handling** specifically. The review's verdict is blunt: the entire boost economy is **scaffolding with no signal through it** (IBEX-027, confirmed H — "inert end-to-end"), and factory usage is absent entirely (`todo.md:12`). Meanwhile the prior-art digest is unanimous that dominant bots run an **always-on mineral→reaction→boost pipeline** feeding both war and GCL ("Dominant bots never fight or upgrade unboosted at end-game"; KasamiBot's creeps *request* boosts from a labmanager; the XGH2O GCL-temple doctrine; Tigga: "the cost in boosts to crack this thing far outweighs the cost in energy to build it" — i.e. boosts are the currency of both siege and siege-defense). A bot without this pipeline plateaus at RCL8 and loses any boosted engagement.

### What EXISTS and works (do not rebuild)

- **A real per-room reaction runner.** `LabsMission` (`missions/labs.rs`, 794 lines) is a live FSM (`Idle → RunReaction/RunReverseReaction → Wait(20) → Idle`) launched per owned room by `ColonyMission` when labs exist (`missions/colony.rs:259-267`, `labs.rs:742-748`). It:
  - selects **input labs** as those within range 2 of *all* labs and uses the rest as outputs (`labs.rs:128-146`) — satisfied by the foreman lab stamp, whose 2 "source" labs are within range 2 of all 8 reaction labs by construction (`screeps-foreman/src/stamps/lab.rs:26-95`);
  - picks a target reaction greedily: walk `desired_resources()` (all 30 T1/T2/T3 compounds, `labs.rs:198-240`), each with a flat 10,000 target (`get_desired_storage_amount`, `missions/constants.rs:3-8`), recursively expanding missing components and choosing the reaction with the most runnable volume (`labs.rs:242-349`);
  - runs `run_reaction` across all output labs respecting per-lab cooldown and `LAB_REACTION_AMOUNT` (`labs.rs:493-521`), with a symmetric reverse-reaction state that decomposes only excess-over-desired stock (`labs.rs:336-342, 527-711`);
  - **wires lab logistics correctly through the transfer system**: deposit requests for input reagents, withdraw requests for outputs/unwanted minerals, as `TransferType::Haul` generators (`labs.rs:70-100, 392-471, 542-638`). This is exactly the [0007](0007-hauling-logistics.md) seam — lab hauling needs no new mechanism.
- **Mineral supply.** `MineralMiningMission` mines the room mineral via extractor into containers (`missions/localsupply/mineral_mining.rs:19-30, 189-273`); minerals flow to storage/terminal through the same transfer system.
- **Terminal sell/buy thresholds.** `TerminalMission` imports a resource toward `desired_storage_amount + reserve` and sells tiered excess (passive order / active deal) above it (`missions/terminal.rs:100-126, 210-303`); it can passive-buy **base minerals** behind the `market.buy_minerals` feature flag — default **off** (`terminal.rs:86-98`, `features.rs:135-153`).
- **Power processing.** `PowerSpawnMission` keeps the power spawn fed and calls `process_power` (`missions/powerspawn.rs:87-176`) — the GPL sink exists; it is not this ADR's subject but shares the haul seam.

### What is DEAD (the IBEX-027 surface, verified against current code)

- **`BoostQueue` has zero producers and zero consumers.** It is inserted as a world resource (`game_loop.rs:606`) and threaded into every mission's system data (`missions/missionsystem.rs:36/61/187/242`), but a crate-wide search finds **no call site** for `request` / `mark_ready` / `is_ready` / `pending_requests` / `clear` outside `military/boostqueue.rs` itself. It is never even cleared per tick despite its "rebuilt each tick" doc comment (`boostqueue.rs:56-58`).
- **No code in the entire bot issues a `boost_creep` intent.** A crate-wide grep for `boost_creep`/`unboost` finds only a comment (`military/bodies.rs:5`). Even a populated queue would boost nothing.
- **Boosted compositions are unreachable.** `BodyType::Boosted*` variants and their `required_boosts()` tables exist (`military/composition.rs:32-36, 124-150`; T3 constants at `military/bodies.rs:513-526`), and `SquadComposition::required_boosts()` aggregates them (`composition.rs:551-564`) — but no force plan ever selects a Boosted* body (`operations/attack.rs:161-251` picks only unboosted compositions), so the aggregator has zero callers (review §1 IBEX-027 block).
- **`available_boosts` is always empty.** `RoomEconomyData.available_boosts` is initialized to `HashMap::new()` and never populated (`military/economy.rs:48-50, 351`), so `EconomySnapshot::has_boost`/`total_boost` (`economy.rs:127-140`) statically return false/0. `AttackPhase::Prepare`'s doc says "Queue boost production" (`operations/attack.rs:25-26`) but the phase only waits on energy/spawn capacity.
- **Factory: nothing at all.** `StructureFactory` is plumbed as a `TransferTarget` (`transfer/transfersystem.rs:111, 142, 161, 206, 245, 404-407`) and collected in room data (`room/data.rs:538, 674`), and nuke defense would protect it (`missions/nuke_defense.rs:183`) — but no mission produces, no `produce` intent is ever issued, and no commodity logic exists. RCL7 grants the structure (`screeps-common\lib\constants.js:230`: factory `{7:1, 8:1}`).

**Net:** Ibex has a working *reaction* layer with no *demand* signal (it brews 10k of everything, for no one) and no *consumption* layer (nothing boosts, nothing produces commodities). The pipeline is a bridge built from both banks that never meets in the middle.

### Engine ground truth this design must respect (cited; spot-verified where load-bearing)

- **Reactions:** output lab must be within Chebyshev range 2 of both inputs; inputs need ≥ `LAB_REACTION_AMOUNT(5)`; produces 5 (+`PWR_OPERATE_LAB` effect up to 15); output lab cooldown = `REACTION_TIME[product]` (`engine\processor\intents\labs\run-reaction.js:12-16, 26, 38, 56-57`). Lab capacity 3000 mineral + 2000 energy (`common\lib\constants.js:275-276`). Lab counts: **3 @RCL6, 6 @RCL7, 10 @RCL8** (`constants.js:227`).
- **Chain times** (`constants.js:733-768`, spot-verified): OH 20, ZK/UL/G 5; finals XUH2O/XUHO2/XKH2O/XKHO2/XLHO2/XZHO2 60, XLH2O 65, XGH2O 80, XGHO2 150, XZH2O 160; GHO2 30, ZH2O 40.
- **boostCreep:** 30 mineral + 20 energy **per part**; range ≤1; **rejected while the creep is spawning** (`labs\boost-creep.js:15-25`, spot-verified `:20` `target.spawning → return`); **no owner check and no lab-cooldown check** (a reacting lab can boost; allies can be boosted); TOUGH boosts head-first, everything else tail-first (`:33-35`); partial boosts run until the lab empties (`:41-46`). Multiple labs can each boost the same creep in the same tick (one intent per lab).
- **unboostCreep:** strips ALL boosts at once; refund `LAB_UNBOOST_MINERAL(15)`/part (50%) **dropped on the ground**, energy refund 0; lab cooldown = Σ parts × chain-time × 3 — **675 ticks per XGHO2 part** (`labs\unboost-creep.js:32-48`; `constants.js:281-282`; `utils.js:665-669`). Dropped resources decay `ceil(amount/1000)`/tick (`energy\tick.js:12`).
- **renewCreep strips ALL boosts with zero refund** (`spawns\renew-creep.js:45-53`). **recycleCreep** refunds, per boosted part, `30 × ttl/lifeTime` mineral + `20 × ttl/lifeTime` energy, deposited into a container on the creep's tile if available (`spawns\recycle-creep.js:22`; `creeps\_die.js:39-94`).
- **Boost effects** (`constants.js:617-731`): T3 — attack ×4 (XUH2O), ranged ×4 (XKHO2), heal ×4 (XLHO2), fatigue ×4 (XZHO2), tough damage-taken ×0.3 (XGHO2), upgradeController ×2 (XGH2O), dismantle ×4 (XZH2O), build/repair ×2 (XLH2O), carry ×4 (XKH2O).
- **RCL8 GCL doctrine:** upgrade energy is capped at 15/tick room-wide, **but GCL credits the boosted amount** (`creeps\upgradeController.js:42-52, 57, 84-86`) — XGH2O doubles GCL per energy at the cap. This is the engine fact behind the GCL-temple meta.
- **Factory:** `produce` checks recipe level; **leveled commodities require an ACTIVE `OPERATE_FACTORY` effect of exactly that level on every produce**, and the first `OPERATE_FACTORY` use **permanently brands the factory** with the power creep's level (`factories\produce.js:9, 21-23`; `power-creeps\usePower.js:259-274`). Level-0 chains need no power: 500 mineral + 200 energy → 100 bar (20cd) and reverse; 600 energy → 50 battery (10cd), 50 battery → 500 energy (`constants.js:1144-1286`). Capacity 50,000 (`constants.js:357`).
- **Transport economics** (for compression decisions): `calcTransactionCost = ceil(amount × (1 − e^(−range/30)))` (`utils.js:644-659`); same formula for `Terminal.send` (`terminal\send.js:19-20`). Nuker needs 5,000 G (`constants.js:348`).
- **Hard constraints** (rewrite grounding): tick-snapshot concurrency (no same-tick reaction to enemies → boosts must be **stockpiled predictively**, never brewed reactively); single-threaded WASM; CPU = execution + intents (~0.2/intent — reactions and boosts are cheap intents, the *planning* is the CPU); VM-reset resilience; finite creep lifetime (1500t) — which interacts brutally with chain times, below.

### The arithmetic that forces the doctrine (why always-on stockpile, not on-demand)

Per 5 units of final compound, one lab is busy for the sum of the chain's `REACTION_TIME`s. Lab-ticks **per unit**: XKHO2/XLHO2/XZHO2/XUH2O = (10+20+5+60)/5 = **19**; XGH2O = (5+5+5+10+20+15+80)/5 = **28**; XGHO2 = (5+5+5+10+20+30+150)/5 = **45**. One T3 part costs 30 units. So with the full RCL8 cluster (8 working output labs, perfectly pipelined):

- One `BoostedQuadMember` (36 boosted parts: 6 TOUGH + 10 RA + 10 HEAL + 10 MOVE, `composition.rs:127-132`) = 1,080 compound units ≈ 25,200 lab-ticks ≈ **3,150 ticks of an entire room's cluster — more than two creep lifetimes for ONE creep's boosts.**
- A defense floor of 3,000 units × 4 defense compounds ≈ 306,000 lab-ticks ≈ **38 kiloticks of one cluster** (parallelizable across rooms by compound/stage).

**Conclusion (decision-bearing):** boosts can never be produced inside an `AttackPhase::Prepare` window or a defense response. The only viable model is the prior-art one: **produce continuously in peacetime into an empire stockpile; serve boost requests from stock in O(haul) time; gate offense on stock, never gate defense on anything.** The current `LabsMission` already runs "always-on" — what is missing is *direction* (what to brew, empire-wide) and *consumption* (boosting, factory).

## Decision

**Resolve IBEX-027 as WIRE, not delete** — with this design. (Rationale against delete: the review calls the gap "a strategic capability gap"; prior-art synthesis item 6 makes an always-on boost pipeline table-stakes; the competitive analysis lists it as gap G5; and ~80% of the machinery — reaction runner, haul seams, mineral mining, compositions — already exists. Deleting would discard working code to avoid writing the ~20% connective tissue. [0008](0008-combat-and-squad-architecture.md) Migration Step 4 already reserved the wire-or-delete decision point; this ADR is that step, fully designed.)

Adopt a **three-layer pipeline** with hard seams, mirroring the operation/mission/manager layering of [0008](0008-combat-and-squad-architecture.md):

```
Empire ReagentPlanner (NEW, global, cadenced, sheddable)
        │  per-room ReactionAssignment + import/export/buy requests
        ▼
Per-room LabsMission (REWRITTEN KERNEL, same FSM + transfer seams)   FactoryMission (NEW, small)
        │  BoostStation sub-state: load labs for a BoostRequest
        ▼
BoostQueue (WIRED): spawn-orchestrator reservations (0011 D9) → station fulfills → is_ready gates deploy
```

### 1. Empire ReagentPlanner — the demand signal (replaces the flat 10k-of-everything target)

A global resource + system, running at a coarse cadence (~100 ticks; governor-sheddable per [0004](0004-cpu-governance-and-load-shedding.md)). It is a **pure function** of `(empire stocks, threat levels, active objectives, room lab inventories) → (per-compound targets, per-room ReactionAssignments, transfer/market requests)` — host-target testable per [0006](0006-eval-and-iteration-harness.md).

**Stockpile targets (the doctrine), per tier — replaces `get_desired_storage_amount`'s flat `_ => 10_000` (`missions/constants.rs:6`) for compounds:**

| Tier | Compounds | Target | Why |
|---|---|---|---|
| **A — Defense floor (never below, all threat levels)** | XGHO2, XLHO2, XZHO2, XKHO2 | **3,000 each per war-capable RCL7+ room** (= one lab fill = 100 boosted parts each) | Tigga's defense thesis: boosted TOUGH/heal is what makes ramparts unbreakable. Production priority over everything when below floor. Stock physically distributed so each room can boost defenders without a terminal hop. Scales +3,000/compound per step of sustained `ThreatLevel ≥ PlayerRaid` (`military/threatmap.rs:38-53`). |
| **B — Economy (when A is full)** | XGH2O | rolling **6,000 per RCL8 room** (≈ 200 WORK-parts) | The engine GCL fact above: ×2 GCL per energy at the 15/t cap (`upgradeController.js:84-86`). This is the highest-ROI peacetime sink for surplus lab time and the prior-art consensus (GCL-temple doctrine). |
| **C — Offense (war footing or sustained surplus)** | XUH2O, XZH2O, + deepen A stocks | sized from `WarOperation` posture: planned campaign force-plans × `required_boosts()` × 1.5 waves | Produced ahead of campaigns by the planner reading declared (not yet active) objectives — predictive per the tick-concurrency constraint. XZH2O's 160t chain (`constants.js:758`) makes dismantle the slowest stock to build: start it first or buy it (ADR 0012). |
| **D — Strategic intermediates** | G (+ GH, OH buffers) | 5,000 G per nuker (`constants.js:348`) + 1,000 OH buffer per active cluster | G is the shared trunk of XGH2O/XGHO2 chains and the nuke payload; OH is in every T2/T3 chain (`constants.js:484-615`). |

**Input sourcing, in priority order:** (1) own room minerals (`MineralMiningMission`, already live); (2) **inter-room terminal balancing** — the planner emits import requests that ride `TerminalMission`'s existing threshold machinery (`terminal.rs:210-303`), fixing its `//TODO: Include resources that are requested by transport system but don't exist in the room` (`terminal.rs:204`); (3) **market buy** of missing base minerals (especially Catalyst, which Ibex rooms may simply not have) through ADR 0012's trust-guarded order path (`terminal.rs:86-98` already enumerates exactly the right base-mineral set behind `buy_minerals`) — competitive-analysis G6's "auto-buy missing minerals on a credit threshold". Buying **intermediates** (OH, GH, even T2) off-market is permitted under ADR 0012 price guards — prior art says buying mid-chain is often correct (factories digest; same logic applies to lab chains).

**Reaction scheduling:** the planner assigns each room cluster one **batch** = `min(3000, remaining target)` of one compound, deepest-deficit-weighted, chain-stage-aware (a room is assigned a stage only if its inputs are stocked or inbound). The per-room `LabsMission` keeps its FSM shape (batch → Idle → next assignment) but takes its target from the assignment instead of `desired_resources()`/`get_desired_storage_amount` (`labs.rs:154, 260-263`). Minimum batch 500 to amortize haul trips; cooldowns are already respected per-lab (`labs.rs:500-506`). Multi-room parallelism comes free: different rooms brew different stages/compounds and terminals move intermediates — this is what makes Tier A's 38-kilotick single-cluster cost a non-issue empire-wide.

### 2. Lab role assignment per room (rides the [0009](0009-room-planning-and-multiroom-layout.md) stamp)

The foreman stamp's geometry is kept as-is: **2 source/input labs + 8 reaction labs**, all reaction labs within range 2 of both sources (`stamps/lab.rs:74-95`; engine requirement `run-reaction.js:26, 38`). Two refinements:

- **Boost labs are a temporary re-role, not a fixed allocation.** Because `boostCreep` has **no lab-cooldown check** (`boost-creep.js`, verified — no cooldown read), any reaction lab can serve as a boost lab the moment it holds the right compound + energy. A `BoostStation` assignment displaces at most N≤4 reaction labs: haul out their product, haul in compound (30/part) + energy (20/part), boost, haul back, resume reacting. No standing idle "boost lab" tax in peacetime — **except** Tier A: each war-capable room keeps **one designated defense-boost lab pre-loaded** with XGHO2 or XLHO2 (rotating per threat assessment) so the first defense wave boosts with zero haul latency. That lab still reacts; it just isn't drained.
- **The boost tile is plan metadata.** The stamp's diagonal road corridor tile at (1,1) is Chebyshev-adjacent to up to 6 labs — one creep standing there can receive **all four T3 boosts in a single tick** (one `boostCreep` intent per lab, same tick; engine processes each lab's intent independently). [0009](0009-room-planning-and-multiroom-layout.md) records this tile in the room plan so spawn routing (ADR 0011) and the boost job target it without per-tick search.

### 3. Boost-on-spawn — feasibility verdict (engine-checked) and the flow

The competitive analysis (G5) describes Overmind-style "spawn-time boosting (boost while the creep is still spawning)". **Engine verdict: boosting a creep *during* spawn is impossible** — `boostCreep` explicitly rejects `target.spawning` (`labs\boost-creep.js:20`, spot-verified). The *intent* of the idea (minimize unboosted travel, maximize boosted lifetime) survives intact via:

1. **Pre-stage during spawn.** A 36-part body spawns in 108 ticks (`CREEP_SPAWN_TIME=3`/part) — ample time to haul compounds + energy into the station labs concurrently. The spawn orchestrator emits the `BoostRequest` reservation **when it schedules the demand** (ADR 0011 D9), not when the creep emerges.
2. **Emerge → boost tile → boosted in ≤ walk + 1 tick.** ADR 0011 directs the fresh creep to the plan's boost tile as its first order (spawn `directions` biased toward the lab cluster where geometry allows); all loaded labs fire `boostCreep` the tick it arrives. Unboosted overhead ≈ one in-base walk (~10–20 ticks of 1,500), versus Overmind-equivalent outcomes.
3. **`is_ready` is checked before spawning, not after.** The deploy gate (below) means a squad's creeps are never spawned into a station that can't boost them — avoiding the worst case (boosted body with unboosted stats burning TTL waiting on a chain that takes 3,000+ ticks, per the arithmetic above).

### 4. BoostQueue wiring — producers, fulfillers, consumers (the IBEX-027 closure)

The existing API shape (`request`/`mark_ready`/`is_ready`, `boostqueue.rs:80-116`) is kept; its keying and accounting are corrected:

- **Key by `DemandId`, not `Entity`.** `BoostRequest.requester: Entity` (`boostqueue.rs:19-21`) reintroduces exactly the recyclable-index hazard [0001](0001-entity-model.md)/IBEX-002b eliminates; the wired queue keys on the minted **demand id** from ADR 0011 D1 (same [0001](0001-entity-model.md) id discipline). Squad demands carry their `SquadId`-derived `GroupId` for attribution, and **non-squad consumers** — boosted upgraders (gap-analysis G-7a), or any 0011 demand carrying a `boosts` field — file through the same key with no squad machinery at all. (The queue stays an **ephemeral, rebuilt-per-tick resource** as its doc comment intends (`boostqueue.rs:56-58`) — and gains the missing per-tick `clear()` call; nothing about it is serialized, so this is format-safe.)
- **Account energy too.** `amount_needed()` covers only the 30 mineral/part (`boostqueue.rs:39-42`); fulfillment must also reserve 20 energy/part (`constants.js:276-277`) in the station labs.
- **Sole producer: the spawn orchestrator** (ADR 0011 D9 — the reservation is emitted when the demand is scheduled). Owners file *demands*, not boost requests: the Squad Manager ([0008](0008-combat-and-squad-architecture.md)) declares `boosts` on its combat demands from `SquadComposition::required_boosts()` (`composition.rs:551-564`) — finally giving that aggregator its caller — and any other demand owner (e.g. a boosted-upgrader demand, G-7a) does the same. Boosted force plans become *selectable* by the operation layer once `EconomySnapshot.available_boosts` is real (next bullet): the existing `plan_by_detected_threat` (`attack.rs:208-251`) upgrades a composition to its Boosted* variant **iff** `economy.has_boost(...)` covers it from stock.
- **Fulfiller: the room's `LabsMission` BoostStation state**, which converts pending requests into the same `TransferDeposit/WithdrawRequest`s the reaction states already emit (`labs.rs:392-471` is the template) and calls `mark_ready` when labs are loaded. `EconomyAssessmentSystem` populates `available_boosts` from storage+terminal+lab stocks (closing the always-empty map at `economy.rs:351`).
- **Consumers — two gates:**
  - **Offense:** the Squad Manager treats `is_ready` as a precondition of *spawning* the boosted roster ([0008](0008-combat-and-squad-architecture.md)'s deploy gate), and the per-creep boost step as part of Forming. A request unfulfillable from empire stock within a deadline flips the objective's force plan back to the unboosted composition (or `Impossible` for siege plans that require boosts) — never an indefinite Prepare hang (Field Report B discipline).
  - **Defense: boosts NEVER gate defense.** Defense squads spawn immediately at any threat level; if the Tier A pre-loaded lab is ready they boost on the way to the fight (the boost tile is in-base), otherwise they fight unboosted. `BoostPriority::Critical` (`boostqueue.rs:12-13`) preempts any reaction batch: the station displaces reaction labs immediately under `ThreatLevel ≥ PlayerRaid`. Boosts *upgrade* defense; they must never *delay* it.

### 5. End-of-life policy (engine-economics-driven)

- **Never renew a boosted creep** — renew strips all boosts with zero refund (`renew-creep.js:45-53`). Consistent with and reinforcing [0008](0008-combat-and-squad-architecture.md)'s pre-spawn-over-renew and ADR 0011 D8 (the renew-policy owner): renew is orchestrator-internal and legal only for small (≤~15-part) **unboosted** utility creeps at idle spawns — boosted and combat bodies are categorically renew-ineligible, so no squad or demand path can ever strip a boost via renew.
- **Recycle, don't unboost, retiring boosted creeps.** Recycle refunds `30 × ttl/lifeTime` mineral + `20 × ttl/lifeTime` energy per boosted part into a container (`_die.js:39-94`) with **no lab cooldown**; unboost refunds a flat 50% mineral, 0 energy, **dropped on the ground** (decaying `ceil(amount/1000)`/tick), and costs the lab `Σ parts × chain × 3` cooldown — **675 ticks per XGHO2 part** (`unboost-creep.js:47`; `utils.js:665-669`). Break-even is ttl/lifeTime < 50% for mineral only; the lab-cooldown cost makes unboost strictly worse in practice. **Policy: the Squad Manager routes retiring boosted members (objective `Succeeded`, member rotated out with TTL remaining) to recycle at a spawn-adjacent container; `unboostCreep` is not implemented in the initial wire** (revisit only if eval data shows large T3 loads being lost at high TTL — and then at a designated sacrificial lab with a hauler pre-positioned for the ground drop).

### 6. Factory — level-0 only, deliberately

A new **`FactoryMission`**, shaped like `PowerSpawnMission` (`powerspawn.rs` — transfer generator for inputs + a `produce` call in `run_mission`; ~150 lines), launched by `ColonyMission` when a factory exists. The `TransferTarget::Factory` plumbing already exists end-to-end (`transfersystem.rs:111-245`). Scope:

- **Battery energy buffering.** When room storage energy exceeds `2 × get_desired_storage_amount(Energy)` (= 400k, `missions/constants.rs:5`), compress 600 energy → 50 battery (10cd, `constants.js:1144-1286`); decompress when below ½ target. Rationale: 12:1 transport compression for a 16.7% round-trip loss — under `cost = ceil(amount × (1−e^(−d/30)))` (`utils.js:657-659`), batteries cut long-range energy transfer cost ~12× for feeding remote wars, incubated colonies ([0009](0009-room-planning-and-multiroom-layout.md)/Inc 7 incubation), or a future GCL-temple room.
- **Bar compression of surplus minerals** (500 mineral + 200 energy → 100 bar, 20cd) only when the raw mineral exceeds its storage target *and* ADR 0012's price check says bars beat raw on the market; otherwise `TerminalMission` keeps selling raw as today (`terminal.rs:276-301`).
- **Leveled commodity chains: explicitly DEFERRED.** They require an active `OPERATE_FACTORY` effect per produce, and the **first use permanently brands the factory's level** (`produce.js:21-23`; `usePower.js:259-274`) — an irreversible empire decision that belongs with a power-creep ADR, not here. **Invariant: no code path may call `OPERATE_FACTORY` until a deliberate, recorded decision.** Prior art agrees the vertical chains need ~5 leveled factories and that buying intermediates is often better — there is no cheap partial win to grab now.

### 7. CPU posture (per [0004](0004-cpu-governance-and-load-shedding.md))

Intents are cheap: a full cluster issues ≤8 `run_reaction`/tick only on cooldown-expiry ticks, boosting is ≤4 intents per creep once, factory ≤1/cd — all ~0.2 CPU each. The *planner* is the sheddable part: **Conserve** stretches the ReagentPlanner cadence; **Critical** defers it entirely (rooms finish their current batch and idle). Reaction execution and Tier-A defense-boost loading are **proposed** entries in [0004](0004-cpu-governance-and-load-shedding.md)'s shed-tier registry (never-shed/Critical tier; pending 0004 sign-off — additions to that registry require amending 0004), so that a loaded defense-boost lab keeps working at Critical. Lab/factory hauling is ordinary [0007](0007-hauling-logistics.md) traffic, already governed. No pathfinding is introduced anywhere in this pipeline.

## Alternatives Considered

## Consequences

**Positive**
- **IBEX-027 closed as WIRE** with every layer owned: producer (the spawn orchestrator, from demand-declared boosts — ADR 0011 D9), fulfiller (BoostStation), consumers (deploy gate + defense fast-path). `composition.rs:551`'s aggregator and `economy.rs:127-140`'s `has_boost`/`total_boost` gain real callers; `available_boosts` is finally populated.
- **Defense boosts are ALWAYS available** (Tier A floor + pre-loaded lab + Critical preemption + never-gated spawning) — the war-reserve policy the operator asked for, tied to `ThreatLevel`.
- **RCL8 surplus gets compounding sinks** (prior-art synthesis item 10): XGH2O doubles GCL at the cap; batteries make energy mobile; G feeds nukers. The bot stops plateauing.
- **Predictive, not reactive** — stockpile doctrine is the only boost model compatible with tick-snapshot concurrency and 1,500-tick lifetimes (the arithmetic section makes this quantitative, not aspirational).
- **Testable kernels** per [0006](0006-eval-and-iteration-harness.md): the ReagentPlanner decision function, chain-math (targets→assignments), boost-readiness predicate, and end-of-life policy are all pure logic; harness scenarios assert defense-boost latency and that a boosted objective never hangs in Forming.
- **Boosted compositions become reachable**, upgrading [0008](0008-combat-and-squad-architecture.md)'s force plans against boosted opponents (threatmap already assumes T3 enemies, `threatmap.rs:113-133` — today we model their boosts while fielding none of our own).

**Negative / costs**
- A new empire-level system and a tunables surface (tier sizes, batch sizes, cadences, boost-upgrade thresholds). Mitigated: all config-driven and eval-diffable; the planner's pure-function shape keeps tuning out of game code.
- Stockpile capital: ~12k minerals/room (Tier A) + market spend sit idle by design. That *is* the insurance product; the eval harness prices it via the military win-rate term.
- Boosting concentrates value into creeps that can die: a wiped boosted quad member loses ~1,080 T3. Bounded by [0008](0008-combat-and-squad-architecture.md)'s retreat thresholds/cohesion invariant — and is precisely why this ADR sequences **after** Inc 4 cohesion.
- More haul traffic around the lab cluster (station swaps, factory feeds). Ordinary [0007](0007-hauling-logistics.md) load; the stamp's road corridor already serves it.

**New risks**
- **Mis-tiered planner** starving defense floor for economy brewing → the floor is an absolute priority ordering, asserted by a harness invariant (Tier A deficit ⇒ next assignment is Tier A).
- **Market dependence** for Catalyst/missing bases under ADR 0012's trust guards (IBEX-018) — a manipulated market can throttle T3; mitigated by the floor being buildable from any 2 owned mineral types + bought X, and by intermediates-buying as a fallback.
- **Stale assignments after VM reset:** assignments are derivable state — the planner recomputes from stocks on its first post-reset run; per-room batch progress already serializes in `LabsState` (ConvertSaveload, `labs.rs:27-48`). No new reset hazard.

**CPU / tick-safety.** Net negligible execution + small intent counts (above); planning sheddable, execution never-shed only where defense-critical. No new panic surfaces: station/factory states reuse the mission `Result` discipline; no `unwrap` on lab resolution beyond what `labs.rs` already does (and that hardening rides [0003](0003-behavior-modeling.md)/[0005](0005-runtime-and-scheduling-model.md) like every mission).

## Incremental Migration Path

Stable seams: **the `BoostQueue` resource API** (already plumbed into `MissionExecutionSystemData`, `missionsystem.rs:36/61`), **the `LabsMission` FSM + its transfer generators**, and **the Squad Manager's objective reconciler** ([0008](0008-combat-and-squad-architecture.md)). Per the plan's mapping note (rewrite-plan §3: "combat Squad-Manager polish (boost pipeline wire-or-delete)" → **Increment 7**), the main pipeline rides **Inc 7**, with an early defense slice in **Inc 4** and kernel groundwork anytime after **Inc 0**. Never break the running bot mid-increment; the existing LabsMission keeps running unmodified until L2 swaps its demand input.

1. **L0 — Kernel groundwork (after Inc 0; Breaking: None).** Extract the reaction-selection logic (`labs.rs:242-349`) and the new chain-math/targets functions as pure kernels with unit tests ([0006](0006-eval-and-iteration-harness.md)); add the per-tick `BoostQueue::clear()` it always claimed to have (`boostqueue.rs:56-58`); populate `available_boosts` in `EconomyAssessmentSystem` (`economy.rs:351`) from storage/terminal/lab stocks. **Validate:** kernel tests green; `has_boost` returns truth on a fixture; zero behavior change in the harness smoke-run (the data is produced, still unconsumed).
2. **L1 — Defense slice (rides Inc 4, after the Squad Manager exists; Breaking: Behavioral).** Tier A targets only: ReagentPlanner v0 (defense compounds, single-room assignment), the BoostStation state in `LabsMission`, the pre-loaded defense lab, and defense-squad boost-on-the-way (never gated). The orchestrator files `BoostRequest`s keyed by `DemandId` (the Squad Manager declares `boosts` on its demands — ADR 0011 D9); key migration note: the `requester: Entity` field (`boostqueue.rs:19-21`) changes type — **ephemeral resource, no Memory/format impact**. **Validate (harness):** defense scenario at `ThreatLevel::PlayerRaid` — defenders spawn immediately (boost never delays spawn), boost within the latency bound when stock exists, fight unboosted when it doesn't; Tier A floor refills after consumption.
3. **L2 — Empire planner + offense gating (Inc 7; Breaking: Behavioral).** Full tier table; swap `LabsMission`'s demand input from `desired_resources()`/flat-10k to ReactionAssignments (the FSM and transfer generators are unchanged); inter-room reagent balancing through `TerminalMission`; boosted-composition upgrades in force plans gated on `has_boost`; `is_ready` deploy gate with the unboosted-fallback deadline; recycle-not-renew end-of-life for boosted members. **Validate:** harness asserts a boosted objective either deploys boosted or falls back within the deadline (no Forming hang — the Field Report B regression test); cross-room chain staging produces a T3 compound end-to-end; cohesion metric unaffected.
4. **L3 — Market integration (Inc 7, with ADR 0012; Breaking: Behavioral).** Planner emits buy requests for missing bases/intermediates through ADR 0012's trust-guarded path (extends `can_purchase_resource`, `terminal.rs:86-98`); flip `market.buy_minerals` default **on** under the credit-reserve guard (`features.rs:148`) — gated on ADR 0012's M4 adversary scenario passing (its §1 `getHistory` caveat). **Validate:** scenario with a Catalyst-less empire still reaches a full Tier A floor via purchases; spend respects `credit_reserve`.
5. **L4 — FactoryMission (Inc 7, parallel to L3; Breaking: None).** Level-0 battery buffering + price-gated bar compression; `ColonyMission` launch site mirrors `powerspawn` (`colony.rs:246-257`). The `OPERATE_FACTORY` invariant (never call it) is asserted in review + a debug assert. **Validate:** storage-energy oscillation scenario compresses/decompresses across the thresholds; bars sell only when the price check passes; no factory intent issued without inputs.
6. **L5 (optional, evidence-gated) — Unboost recovery.** Only if eval data shows ≥X T3 units/kilotick lost to high-TTL retirement deaths; designated sacrificial lab + pre-positioned hauler. Default: not built.

**Breaking-change summary:** L0 — **None**. L1/L2/L3 — **Behavioral** (what gets brewed/bought/boosted changes; `BoostQueue` is ephemeral and `LabsState`'s serialized shape is untouched — assignments are derivable, not persisted; any later decision to persist planner state must follow [0002](0002-serialization.md)'s `serde(default)` + labelled-segment discipline). L4 — **None** (new mission variant addition follows the same `MissionData` enum-extension path as existing missions; additive). L5 — **Behavioral**. **No Memory/format break and no state drop anywhere in this pillar.**
