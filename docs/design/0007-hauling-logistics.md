## Context

Hauling is the bot's circulatory system: every economic and military intent ultimately depends on a hauler having moved energy to a spawn, extension, tower, controller container, link, or terminal. The review did **not** rate the current matcher a survival risk in isolation (IBEX-030 was adjusted High→Medium, confidence M), but it is one of the **un-budgeted heavy consumers that compose into the Field Report C floor**, and it is the `HaulJob` FSM pilot ([0003](0003-behavior-modeling.md)) and a `&dyn TransferRequestSystemData`-shaped testability win ([0006](0006-eval-and-iteration-harness.md)). This ADR specializes those pillars for the logistics subsystem and decides the hauling-algorithm direction the rewrite converges on. It does **not** re-open the governor or pathfinding-facade decisions (those are [0004](0004-cpu-governance-and-load-shedding.md)); it states how hauling plugs into them.

Hard constraints (from the rewrite grounding): **single-threaded WASM** (no threads/locks/atomics-for-parallelism); **CPU = execution + intents** (`transfer`/`withdraw`/`pickup` each charge ~0.2 CPU when logged, so a hauler that visits more nodes costs more in *both* dimensions); **VM-reset resilience** (the matcher holds **zero** persisted state — see below — which is actually a strength here); **finite creep lifetime** (a hauler lives 1500 ticks, decays, and must be pre-spawned/replaced — the hauler-count model below must plan for replacement, not persistence).

### The current model, precisely

The transfer queue is a **per-tick, lazily-populated supply/demand store** matched by a **greedy per-creep matcher**. The full lifecycle is one tick wide:

1. **Registration (lazy).** Each `RoomTransferMission::pre_run_mission` registers two *generators* per room (`room_transfer.rs:614–624`) rather than eagerly building requests: `register_generator(room, HAUL|USE, …)` and `register_generator(room, HAUL|LINK|USE, …)`. A generator is a `Box<dyn Fn(&dyn TransferRequestSystemData, &mut dyn TransferRequestSystem, RoomName) -> Result<(), String>>` (`transfersystem.rs:1316`) that, when run, enumerates the room's structures/dropped resources and calls `request_withdraw`/`request_deposit` (≈18 call sites in `room_transfer.rs`, lines 242–583).
2. **Flush-on-first-query.** `LazyTransferQueueRooms::get_room` (`transfersystem.rs:1358–1367`) calls `flush_generators`, which **drains every generator whose `transfer_types` intersect the query** via `swap_remove` (`transfersystem.rs:1339–1356`) before any read. So the first consumer to touch a room pays the full generation cost for that room; later consumers in the same tick read the already-materialized `TransferQueueRoomData`. This is the IBEX-031 mechanism — and it is **reservation-safe** (below), so the residual concern is only testability/clarity (the `//TODO` at `transfersystem.rs:1330`).
3. **Reservation accounting.** Each room holds `withdrawls: HashMap<TransferWithdrawlKey, u32>` and `pending_withdrawls` (the mirror for deposits). `get_available_withdrawl(key) = (withdrawl − pending_withdrawl).max(0)` (`transfersystem.rs:483–485`; deposits symmetric at `:495–497`). When a hauler is matched, `register_pickup`/`register_delivery` (`transfersystem.rs:554–584`) bump `pending_*`, so **the next hauler in the same tick sees reduced availability regardless of match order** — this is exactly why IBEX-031's nondeterminism claim was REFUTED.
4. **Match (greedy, per idle hauler).** An idle hauler calls `get_new_pickup_and_delivery_full_capacity_state` → `select_pickup_and_delivery` (`haulbehavior.rs:206`, `transfersystem.rs:2113`). It picks **one** (pickup, delivery) pair, reserves it, optionally chains nearby same-route deliveries (`get_additional_deliveries`, `haulbehavior.rs:251–365`), and stores the resulting tickets in the creep's FSM state.
5. **Execution & teardown.** `tick_pickup`/`tick_delivery` (`haulbehavior.rs:409–567`) drive the move/withdraw/transfer intents. At end of tick `TransferQueueUpdateSystem::run` calls `transfer_queue.clear()` (`transfersystem.rs:2436–2438`) — **the entire queue, generators and materialized rooms, is discarded and rebuilt next tick.** There is no cross-tick amortization of the matching work.

### The matcher's real complexity

Define: `R_p` = pickup rooms, `R_d` = delivery rooms, `N` = nodes (targets) per room, `K` = distinct withdrawl/deposit keys per node (resource × priority × transfer-type), `P` = priority levels (= 4: High/Medium/Low/None), `H` = idle haulers re-matching this tick.

**Priority fan-out.** `select_pickup_and_delivery` loops the pairs from `generate_active_priorities(allowed, allowed)` (`utility.rs:85`). That generator interleaves a Pickup and a Delivery state, walking `High→Medium→Low→None` on each side and yielding a `(pickup_mask, delivery_priorities)` or `(pickup_priorities, delivery_mask)` pair per accepted level (`utility.rs:34–82`). Upper bound **≈ 2·P = 8 pairs**, and it short-circuits the moment one pair yields a match (`transfersystem.rs:2129–2143`).

**Per priority pair → `select_best_delivery` (`transfersystem.rs:1763–1844`):**
- `get_available_withdrawl_totals_by_priority` scans every pickup room's `withdrawl_resource_stats` → **O(R_p · N · K)** (`:1782`).
- `select_deliveries` iterates `R_d` rooms × `N` nodes, calling `node.select_delivery` on each. `select_delivery` (`transfersystem.rs:677–770`) is **O(available_resources · K)** per node. So `select_deliveries` is **O(R_d · N · K · |resources|)** (`:1789`).
- Then, **for each returned delivery**, it re-runs `select_pickups` (`:1811`), which iterates `R_p` rooms × `N` nodes calling `node.select_pickup` (`transfersystem.rs:586–675`, itself **O(|desired| · K)**). That nests as **O(D · R_p · N · K)** where `D` = deliveries returned (`≤ R_d · N`).

Composing: a single `select_pickup_and_delivery` is bounded by roughly

> **O( P · R_d · N · ( K + R_p · N · K ) ) = O( P · R_p · R_d · N² · K )**

and the whole-tick cost is that **× H** idle haulers, with **no `can_execute_cpu`/governor gate anywhere on the path** (the IBEX-030 finding). The `N²` term is the pickup-per-delivery re-run; the `×H` is the un-amortized per-creep re-match.

**Why the report bounds it to Medium, not High.** In practice the multipliers are small: a haul mission passes `pickup_rooms = &[self.room_data]` (a **single** room, `haul.rs:289`) and `delivery_rooms` = the home rooms (typically 1), so `R_p ≈ R_d ≈ 1`. Per-room node data is generated **once per tick** and cached (the `swap_remove` flush), so the generator cost is paid once, not `×H`. `N` is the structure/dropped-resource count of one room (tens, not the "20 rooms × 100 targets" cross-product the seed feared). So the *typical* cost is `H · P · N² · K` with `R=1` — real, ungated, per-creep, but bounded by per-hauler scope and once-per-tick generation. The danger is the **shape**: it scales with idle-hauler count and room node density with no circuit-breaker, and it runs on the post-reset tick (when generators are cold and the bucket is starved) — i.e. it is a Field Report C *contributor*, not the dominant term. The `value = resources / (pickup_len + delivery_len)` ranking (`transfersystem.rs:1837`) uses **`get_range_to` (linear/Chebyshev), not pathfinding**, and `find_nearest_linear_by` (`findnearest.rs:153`) is likewise linear — so the matcher itself issues **no PathFinder ops**; its cost is pure iteration + a few intents, which is why it never appears in the pathfinding budget yet still spends execution CPU.

### Three review findings this ADR must honor (do not mis-state)

- **IBEX-030 (Medium):** the matching cost above is real and ungated, but **bounded by small per-hauler room scope** and once-per-tick caching. The remedy is a bucket gate + re-match cadence + a per-tick supply snapshot — **not** a rewrite of the cost model into something heavier.
- **IBEX-031 (REFUTED / Low):** lazy generators are **reservation-safe and type-filtered** — they are *not* a source of double-serve or nondeterminism. `flush_generators` drains all intersecting generators before any read; node-level type filtering (`select_*`'s `matches`, `transfersystem.rs:430/445`) means a HAUL consumer never sees TERMINAL-only requests; `pending_*` reservation makes a second hauler see reduced availability. **Do not re-flag lazy flushing as a correctness bug.** The only residual is testability + the viz-on `flush_all_generators` (`transfersystem.rs:2410`) diverging from the lazy matcher view.
- **IBEX-011 (REFUTED as data loss / Low-Open):** partial hauls do **not** strand or lose resources. `tick_delivery` keeps carried resources in the creep; `tickets.remove(0)` (`haulbehavior.rs:554`) fires only for *invalid/exhausted* targets; on a rejected transfer (`transfered == false`) it returns `Some(next_state())` and the creep re-plans against its **current** store (`haulbehavior.rs:558–566`). The genuine defect is **commitment churn**: a hauler can abandon an in-progress delivery plan and re-enter Idle re-selection (wasted ticks, target switching), plus `consume_deposit` precedes the transfer intent so a rejected transfer transiently mis-accounts the ticket (recovered on re-plan). The fix is a *committed-delivery guard*, not a data-recovery mechanism.
- **IBEX-050 (tech-debt):** the `get_capacity − Σ get_used_capacity(Some(r))` workaround for the upstream `get_used_capacity(None)` double-count bug is **copy-pasted ≥6 times** (`haulbehavior.rs:28,80,388,479`; `haul.rs:168`-region; commented-out correct form alongside each). All sites must change together when the API bug is fixed — a single `creep_used_capacity`/`creep_free_capacity` helper removes the hazard.

## Decision

**Keep the lazy-generator supply/demand store and the per-creep greedy matcher as the substrate. Specialize it along five concrete, CPU-bounded axes, landing each behind a stable seam.** The matcher is *good enough* (the report's verdict); the rewrite makes it **CPU-bounded, bucket-aware, re-decision-cadenced, commitment-stable, and route-aware in its sizing**, and explicitly declines the Overmind full logistics-network rewrite as over-scoped for the measured pain. Concretely:

1. **Two-phase snapshot-then-assign behind a frozen seam (closes IBEX-030's shape; testability for IBEX-031).** Make the once-per-tick generation explicit: at the top of the hauling pass, flush all relevant rooms **once** into an immutable per-tick `TransferSnapshot` (the `flush_all_generators` path already exists for viz, `transfersystem.rs:1399`), then have every idle hauler assign against that snapshot, mutating only the cheap `pending_*` reservation maps. This is the review §9 / IBEX-030 recommendation ("snapshot supply/demand once, then assign creeps against the cache") and it makes the matcher a **pure function of (snapshot, creep) → (pickup, delivery)** that is host-target testable against in-memory fixtures (the `&dyn TransferRequestSystemData` seam is already the right shape — [0006](0006-eval-and-iteration-harness.md)). Asymptotic cost is unchanged per hauler, but generation is provably paid once and the matcher becomes replay-diffable (the IBEX-031 testability residual). **Seam:** `select_pickup_and_delivery(&snapshot, creep)`.

2. **Bucket gate + re-match cadence (the core IBEX-030 remediation; ties [0004](0004-cpu-governance-and-load-shedding.md)).** Hauling itself is **never shed** ([0004](0004-cpu-governance-and-load-shedding.md) pins defense/spawn/**haul**/movement/serialize as always-on), but the *re-decision* is sheddable: a hauler **re-runs the matcher only when it has no committed plan, on completion, or every N ticks** — not unconditionally per idle tick. Under the governor's **Conserve** tier, raise the re-match cadence (IBEX-030's "re-decide only every N ticks"); under **Critical**, haulers ride their existing tickets and skip re-selection entirely. The matcher's cost is execution-only (no pathfinding ops), so it is gated by the governor's tier read, not the pathfinding facade — but the *movement* a haul plan induces draws from the shared per-tick ops pool ([0004](0004-cpu-governance-and-load-shedding.md)) like every other path. **The hauler keeps moving at Critical** (it has a committed plan), satisfying the "never fully freeze" invariant.

3. **Committed-delivery guard (closes IBEX-011 churn; `todo.md:3`).** Once a hauler has begun depositing a ticket set, it **finishes that set before Idle can re-plan** — prefer *confirm-then-consume* (only `consume_deposit` after the transfer intent is accepted) so a rejected transfer does not mis-account, and a "committed" flag on the FSM state so damage/partial-haul does not bounce the creep back to harvesting (the literal `todo.md:3` item). This is a **behavioral** change folded into the [0003](0003-behavior-modeling.md) HaulJob FSM pilot: the FSM owns the commitment invariant, and the [0003](0003-behavior-modeling.md) "one guarded intent sink" already prevents the double-fire the confirm-then-consume reorder might otherwise risk.

4. **Hauler count from route distance + shared predicted capacity (`todo.md:11`, `todo.md:22`).** Two coupled sizing fixes:
   - **Route-distance sizing.** Today the count derives from **Manhattan room distance**: `room_offset_distance = home_room_data.name − room_data.name`, `room_manhattan_distance = |Δx| + |Δy|` (`haul.rs:199–201`), feeding `range_multiplier = 1/((max_distance·2)+1)` and `max_haulers = 3 + max_distance·3` (`haul.rs:264–267`). The `//TODO: Use find route plus cache` at `haul.rs:195` flags this. Replace `max_distance` with the **route distance** from `RoomRouteCache::get_route_distance` (the same cache war already uses, budgeted through the [0004](0004-cpu-governance-and-load-shedding.md) facade so the cold-cache `find_route` storm is governed), so a route-blind estimate (linearly-close but behind hostiles / across SK rooms) no longer over- or under-provisions haulers. This mirrors the IBEX-032 linear-vs-route fix in expansion.
   - **Shared predicted storage capacity per tick (`todo.md:22`).** A per-tick `predicted_capacity` for each delivery target — decremented as each hauler reserves against it — lets a hauler **not waste its final delivery tick** waiting on a target it can already see will be full/empty (`tick_delivery`'s `//NOTE: … Needs predicted storage that can be shared across systems`, `haulbehavior.rs:560`). This is the deposit-side analog of the existing `pending_*` reservation, computed once in the Phase-1 snapshot and shared. It is the closest this ADR gets to Overmind's `predictedRequestAmount`/`nextAvailability` coordination — adopted **selectively** for the one place the current code explicitly wants it, not as a whole network.

5. **Single capacity helper + the IBEX-010 `panic!`→`Err` boundary (tech-debt + tick-safety).** Extract one `creep_used_capacity`/`creep_free_capacity` helper (closes IBEX-050; one definition, grep-able) so the upstream double-count fix is a one-line change. Independently, the Nuker-withdraw `panic!` (`transfersystem.rs:208`, IBEX-010) becomes `Err(ErrorCode::InvalidArgs)` + one-shot log; the longer-term type-split of `TransferTarget` into withdraw-capable vs deposit-capable enums (the `//TODO` at `:207/:248`) makes the invalid pairing **unrepresentable** — this is owned by [0005](0005-runtime-and-scheduling-model.md) but called out here because the offending arms live in this file.

**Explicitly declined:** a full Overmind-style logistics network (one universal `transport` role + Gale–Shapley stable matching + dq/dt-with-buffer ranking + `nextAvailability`/`predictedCarry` coordination). See Alternatives — it is a net win on creep count (~30% fewer per Overmind) but is a **large, spiky-CPU** rewrite ([0004](0004-cpu-governance-and-load-shedding.md) would have to budget for synchronized colony re-matchings) that the review's Medium-severity finding does not justify ahead of the survival-class increments. We adopt its *ideas* (signed amount + multiplier already exist as priority; predicted capacity = item 4; snapshot-then-match = item 1) without its global re-architecture.

## Why logistics assignment is hard (so the bound matters)

Optimal multi-pickup/multi-dropoff hauling is the **Vehicle Routing Problem with Pickup & Delivery (VRPPD)**, a generalization of the Traveling Salesman Problem; exact assignment is **NP-hard** and the exact DP for one vehicle's pickup-delivery tour is **O(n²·2ⁿ)** — infeasible under a per-tick CPU budget (Overmind's "Hauling is (NP-)hard"; [`../references/external-references.md`](../references/external-references.md)). Every viable bot therefore runs an **approximation**. Ibex's current approximation is **greedy + reservation**: each hauler greedily maximizes `resources/(pickup_dist + delivery_dist)` over the priority-ordered request set, and the `pending_*` reservation makes the sequence of greedy picks **mutually consistent within a tick** (no two haulers double-commit the same unit). Greedy is the cheapest approximation (`O(n²)` at worst per the bound above, `O(H·N²·K)` in practice) and sacrifices global optimality — it can make a locally-good pick that blocks a better global pairing. Overmind's **Gale–Shapley stable matching** is a strictly better `O(n²)` approximation (a "symmetrically greedy" match that avoids the worst greedy pairings) but does not change the asymptotic class and adds the machinery cost above. **The decision to keep greedy is a deliberate trade**: the report measured the matcher as adequate, the per-hauler scope is tiny, and the dominant survival lever is *bounding* the work (items 1–2), not improving the approximation's optimality. If the eval harness later shows the greedy approximation losing meaningful throughput at scale, item 1's frozen `(snapshot, creep)` seam is exactly where a stable-matching assigner drops in without touching callers.

## Alternatives Considered

| Option | Pros | Cons |
|---|---|---|
| **Keep greedy + reservation; specialize (snapshot-then-assign, bucket gate + cadence, committed-delivery guard, route-distance + predicted capacity, helper)** — chosen | Minimal, incremental, behind stable seams; closes IBEX-030's *shape* + IBEX-011 churn + IBEX-050 + IBEX-010 with no format rewrite; matcher becomes a pure, replay-diffable function; honors the report's "adequate matcher" verdict; CPU-bounded and bucket-aware | Still a greedy approximation (no global-optimality guarantee); predicted-capacity coordination is partial, not network-wide; route-distance sizing depends on the [0004](0004-cpu-governance-and-load-shedding.md) facade landing first |
| **Overmind-style logistics network** (universal transport role, Gale–Shapley `O(n²)`, dq/dt-with-buffer ranking, `nextAvailability`/`predictedCarry`/`predictedRequestAmount`) | ~30% fewer creeps at ~equal CPU (Overmind-measured); better approximation; coordination-free load balancing | Large rewrite; **spiky CPU** on re-match ticks ([0004](0004-cpu-governance-and-load-shedding.md) must stagger colony matchings); unjustified ahead of survival increments by a Medium finding; discards a working, reservation-safe matcher |
| **Pure per-tick greedy, no reservation** (today minus `pending_*`) | Simplest | Two haulers double-commit the same unit within a tick → wasted trips; the `pending_*` accounting is precisely what makes IBEX-031 safe — removing it *re-introduces* the bug the report refuted |
| **Optimal/ILP assignment (Hungarian / min-cost-flow per tick)** | Globally optimal per snapshot | `O(n³)`+ per solve; pathological under the bucket; over-engineered for tens of nodes; no resumability story under VM reset |
| **Persist the matcher state across ticks** (cache assignments in segments) | Avoids per-tick re-generation | The queue is intentionally ephemeral and cheap to rebuild; persisting it adds serialization cost + staleness + a new format-break surface, and IBEX-049 already shows per-creep path serialization is a *cost*, not a win — keep the matcher stateless, persist nothing |

## Consequences

**Positive.**
- IBEX-030's risky *shape* is bounded: the matcher gains a governor tier read + re-match cadence + committed plans, so per-creep re-matching no longer scales unbounded with idle-hauler count, and it is provably gated under Conserve/Critical while hauling itself never sheds (the colony keeps moving energy under CPU pressure — directly the §1 colony-health SURVIVAL term).
- The two-phase snapshot makes the matcher a **pure, host-target-testable, replay-diffable function** — it slots straight into [0006](0006-eval-and-iteration-harness.md)'s `&dyn TransferRequestSystemData` fixtures and is the natural replay-intent-diff target alongside the [0003](0003-behavior-modeling.md) HaulJob FSM pilot. The IBEX-031 testability residual is closed; the lazy-flush contract is documented.
- IBEX-011 churn ends: a committed delivery finishes its ticket set, so damage/partial-haul no longer bounces the hauler back to harvesting (`todo.md:3`), and confirm-then-consume removes the transient ticket mis-accounting.
- Route-distance sizing (`todo.md:11`) stops over-/under-provisioning remote haulers, and shared predicted capacity (`todo.md:22`) reclaims the wasted final-delivery tick — both measurable as energy-throughput gains in the colony-health ECONOMIC term.
- IBEX-050's duplicated capacity workaround collapses to one helper; IBEX-010's reachable tick-aborting `panic!` becomes a logged `Err`.

**Negative / costs.**
- More moving parts in a subsystem the report called adequate: a snapshot lifecycle, a cadence tunable, a commitment flag, and a predicted-capacity map. Each is small but adds surface; the eval harness colony-health score is the guard against a regression.
- The committed-delivery guard trades *flexibility* for *stability*: a hauler that committed to a now-lower-value delivery will not opportunistically switch. This is the intended behavior (churn was the bug) but could under-serve a suddenly-urgent request; bound the commitment to the current ticket set, not the whole campaign, so re-evaluation happens at the next natural decision point.
- Route-distance sizing depends on `RoomRouteCache` being budgeted ([0004](0004-cpu-governance-and-load-shedding.md)); landing it before the facade re-introduces the cold-cache `find_route` cost it is meant to govern — hence the sequencing below gates it on Increment 1.

**New risks / what becomes harder.**
- **Cadence mis-tuning:** too-long a re-match cadence makes haulers slow to react to a new high-priority request (e.g. a tower draining mid-siege). Mitigate by exempting High-priority *new* demand from the cadence (re-match immediately when an ACTIVE-priority deposit appears that no committed hauler serves) and by validating reaction latency in the harness.
- **Predicted-capacity staleness:** the shared per-tick prediction is only as good as the snapshot; a target that fills from a non-hauler source (link, harvester drop) mid-tick can desync. Bounded because the prediction is per-tick (rebuilt from the fresh snapshot) and reservation-style (decrement-only), mirroring the already-safe `pending_*` model.

**CPU / tick-safety.** Net **CPU-positive**: the matcher's execution cost is now gated by the governor tier and amortized by cadence (re-decide every N ticks, not every idle tick), generation is provably once-per-tick, and the matcher issues **no pathfinding ops** (linear ranking) so it does not touch the pathfinding budget — only the haul *movement* does, through the shared pool. Tick-safety improves: the IBEX-010 `panic!`→`Err` removes a reachable whole-tick abort that skips `serialize_world`, and confirm-then-consume narrows the partial-haul mis-accounting window.

## Incremental Migration Path

The stable seam is the **`&dyn TransferRequestSystemData` data interface + the `select_pickup_and_delivery(snapshot, creep)` matcher signature**. The matcher holds no persisted state (the queue is `clear()`-ed every tick), so **every step below is None- or Behavioral-breaking — no serialized format changes, no state drop.** Each step is validated by the eval harness ([0006](0006-eval-and-iteration-harness.md)) before the next; never break the running bot mid-increment. This work **slots into Increment 1 and Increment 6**, riding the governor and the HaulJob FSM pilot respectively — it does **not** introduce a new increment.

1. **Quick-wins (None-breaking, can ship first, independent of any increment).**
   - Replace the Nuker-withdraw `panic!` (`transfersystem.rs:208`) with `Err(ErrorCode::InvalidArgs)` + one-shot log (IBEX-010). **Validate:** unit-test every `(TransferTarget variant, mode)` pair returns `Err` not panic for invalid pairings; harness raid scenario vs an enemy nuker no longer aborts the tick.
   - Extract one `creep_free_capacity` helper, replace the 6+ copies (IBEX-050). **Validate:** grep finds exactly one definition; behavior unchanged (host-target test of the helper at boundary capacities).
   - Document the lazy-flush + reservation contract in-code (close the IBEX-031 testability residual; refute the nondeterminism reading in the comment at `transfersystem.rs:1330`).

2. **Two-phase snapshot-then-assign (Behavioral, None-breaking format) — Increment 1 vicinity, gated on the harness.** Lift generation to one explicit per-tick `TransferSnapshot` (reuse `flush_all_generators`); change `select_pickup_and_delivery` to take `&snapshot`; haulers assign against it, mutating only `pending_*`. **Validate (the IBEX-030/031 gate):** host-target test the pure `(snapshot, creep) → tickets` function against fixtures; **replay intent-diff parity** old-vs-new on a recorded economy-bringup run (this is the same replay gate the [0003](0003-behavior-modeling.md) HaulJob pilot uses) — assert identical pickup/delivery selection; profile under many idle haulers and confirm generation runs once.

3. **Governor tier read + re-match cadence (Behavioral, None-breaking format) — Increment 1, gated on the [0004](0004-cpu-governance-and-load-shedding.md) governor landing.** Add the governor read at the top of the hauling re-decision; re-match only on no-committed-plan / completion / every N ticks; under Conserve raise the cadence, under Critical skip re-selection (ride existing tickets). Exempt newly-appeared High-priority demand from the cadence. **Validate:** induced-CPU-pressure scenario — matcher CPU drops under Conserve/Critical, **haulers keep delivering** (never freeze), and a mid-siege tower-drain is served within the latency bound.

4. **Committed-delivery guard + confirm-then-consume (Behavioral) — Increment 6, folded into the HaulJob FSM pilot ([0003](0003-behavior-modeling.md)).** Add a "committed" FSM flag so a started delivery finishes its ticket set before Idle can re-plan (`todo.md:3`); reorder to confirm-then-consume so a rejected transfer does not mis-account (`haulbehavior.rs:539–552`). **Validate (the IBEX-011 gate):** damage a hauler mid-delivery and assert it finishes the **same** target set rather than returning to harvest; assert no transient ticket mis-account on a rejected transfer; replay intent-diff parity on HaulJob.
