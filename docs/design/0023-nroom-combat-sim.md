Status: **Slices S1ŌĆōS5 DONE 2026-06-24** (P-ENGINE substantially complete). Implements ADR 0022 **P-ENGINE** (operator chose the full N-room engine over a movement-only harness). Extends the single-room `screeps-combat-engine` to N rooms so integrated multi-room flee / attack / group-up and the auction's objective-bed composition tournament (ADR 0022 P-AUCTION) are simulatable end-to-end. **Carried follow-ups:** squad **anchor** cross-room + **cross-edge cohesion** ŌĆö **DONE in P-MOVE 2026-06-24** (rover `LocalPathfinder::find_route` made real + `AnchorPath` routes multi-room via the MoveToRoom layer; the squad pre-groups at the exit and crosses as a bloc ŌĆö distinct folded slots + `SimView`-decoupled member moves; a quad crosses a border and reforms a tight box). **P-MOVE+ (2026-06-24):** rover's `LocalPathfinder::search` is now a **true multi-room A*** (over `Position` nodes, per-room grids via `room_callback`) ŌĆö so the per-room MoveToRoom *workarounds* (`in_room_goal` in resolve_move_direction, the anchor's find_route+exit+cross-step, the orchestrator pre-projection) are all **removed**; cross-room routing is native. And the live rover **`MovementSystem` + resolver** (traffic management) is wired into the sim via `resolve_moves_via_system` (`CombatMovementExternal` + `CombatCreepHandle` + a multi-room `CombatWorldCostSource`), so sim ŌēĪ live (one traffic-managed mover). The formation squad (`SimSquad`) is now **cut over** to `resolve_moves_via_system` ŌĆö every member's move is resolved in one traffic-managed pass (a persisted `move_cache` carries the resolver's path-reuse + stuck-escalation across ticks); the folded-slot geometry stays as the *target* layer and the resolver deconflicts the rest (the wall/corridor/cross-edge tests all hold). `ManagedSimSquad` (the kiting self-play squad) is now **also cut over** to `resolve_moves_via_system` ŌĆö the mover gained `Flee` support + a per-request `MovementPriority`. Two fixes this required, both root-caused: (1) **determinism** ŌĆö `resolver.rs::resolve_conflicts` picked the contested-tile winner with `max_by(priority ŌåÆ stuck_ticks)` and **no Handle tie-break**, so a true tie resolved by std-HashMap per-process seed order ŌåÆ seed-flaky moves (the kiting test passed ~50% of runs); fixed with a `Handle` tie-break + `losers.sort()` (deterministic everywhere, helps live too); regression test added. (2) **kiting positioning** ŌĆö with the duo sharing one kite goal, the neutral tie-break parked the *healer* on the forward shooting tile and the *shooter* one tile out of range; fixed by giving **combat creeps `High` priority** so they win the forward tile (the shooter reaches range 3 and focus-fires). Verified: the kite test passes **24/24** across fresh processes (was 0/24 deterministic-fail, ~50/50 flaky). All pathing now goes through rover. Remaining: the single-creep scenario/siege callers still use `resolve_move_direction` (fine ŌĆö no traffic to resolve for one creep); cross-room `Flee`; a dismantle-immune `Core` StructureKind (S4); `focusMax` (S4). ## Architecture finding (what the sim does ŌĆö and doesn't)

> **Update (2026-06-29, post-P-MOVE+):** the bullet below stating the sim "does NOT run the bot's rover/`MovementSystem`/`AnchorPath`" describes the **S1 cut and is now stale.** Since P-MOVE+ (task #30, this doc's status line) the sim **does** run the live rover `MovementSystem` + resolver via `resolve_moves_via_system` to produce move directions. What remains true is narrower: the sim never *measures* rover quality (route optimality / fatigue / congestion / ops) ŌĆö it scores combat. That measurement gap is [ADR 0033](0033-rover-pathing-sim-and-benchmark.md), which also **extracts** this engine's combat-agnostic movement mechanism (world / terrain / body / contention / fatigue / edge-exit / recording) into a new shared `screeps-sim-core` crate both sims depend on, landing two fidelity fixes there (roads + loaded-CARRY). After that extraction `screeps-combat-engine` = `screeps-sim-core` + the combat layer (`CombatState` + `resolve_combat_tick`), and the engine's movement types are renamed `Combat*`ŌåÆ`Sim*` (`CombatWorld`ŌåÆ`SimWorld`, `CombatTerrain`ŌåÆ`SimTerrain`, `resolve_tick`ŌåÆ`resolve_combat_tick`); the movement claims below hold unchanged, just from the lower crate under the new names.

Verified by reading the crate:

- **The engine is a deterministic tick over a `CombatWorld`.** `resolve.rs::resolve_tick` takes an `Intents` (per-creep combat actions **+ move `Direction`s**, tower actions) and resolves it (two-phase accumulate-then-apply combat + the `movement.rs` same-tile-contention port). It does **not** pathfind.
- **Move *directions* are produced by the agent/harness layer** (`screeps-combat-agent`) from the bot's decision output (`decide_combat`/`decide_squad`) ŌĆö the engine just resolves the given `Direction`s.
- **Therefore the sim validates the bot's DECISIONS over a world; it does NOT run the bot's rover/`MovementSystem`/`AnchorPath`.** The anchor multi-room fix (rover `fd977f2`) is validated by rover unit tests + live, **not** here. P-ENGINE's job is multi-room **decision** behavior (when/where to flee, group-up, engage across a border) + the objective-bed combat the auction needs.
- **Single-room assumptions to lift:** `CombatWorld.terrain` is one `CombatTerrain`; `movement.rs::step` returns `None` at room edges ("a later slice"); `rampart_at(x,y)`/contention key by `(x,y)` not `(room,x,y)`; combat range uses `(x,y)`. Positions are **already** `screeps::Position` (room-qualified) ŌĆö the data model is half-ready.
- **Cross-room movement is NOT a move intent (verified against the real engine).** A creep cannot move off a room edge: `move.js:32` rejects the intent and `movement.js:88` clamps to the edge. Crossing is an **automatic edge-exit** (`creeps/tick.js:52-78`): after movement, any non-NPC creep (engine skips users '2'/'3') *standing on* an exit tile is relocated to the adjacent room's mirror tile (`x==0Ōåö49`, `y==0Ōåö49`), applied in the global stage (`global.js:42`). It is same-tick (engine `bulk.update` mutates in place). The first S1 cut got this wrong (modelled crossing as a `step`); corrected ŌĆö see S1 below.

## N-room design

- **World:** `CombatWorld` keeps `terrain: CombatTerrain` as the default/common room terrain (so single-room builders stay unchanged ŌĆö zero churn) **plus** `rooms: HashMap<RoomName, CombatTerrain>` per-room overrides, read via `terrain_for(room) -> &CombatTerrain` (override if present, else the default). No explicit room graph needed ŌĆö adjacency is `Position` world-coord arithmetic (`Position::checked_add((dx,dy))` crosses rooms), and `Position::get_range_to` is already room-aware (Chebyshev over global coords). *(Caveat of the hybrid: the default terrain applies to every override-less room, so a true multi-room scenario with distinct walls per room gives each room its own override via `terrain_mut`.)*
- **Movement:** crossing a room boundary is **not a move intent** ŌĆö the real engine *rejects* an off-edge move (`move.js:32`) / *clamps* it (`movement.js:88`). So `step()` returns `None` at an edge (in-room only); `resolve_moves` + fatigue read `terrain_for(dest.room())`. The boundary cross is a separate **edge-exit relocation** in `resolve_tick`'s Phase D (engine `creeps/tick.js:52-78` + `global.js:42`): a **non-NPC** creep standing on an exit tile is moved to the adjacent room's mirror tile (the tile one `checked_add` step across that edge). It is **same-tick** ŌĆö a creep that *moves* onto the edge crosses that tick (engine `bulk.update` mutates in place, so `tick.js:52` reads the post-move position), so a border cross is effectively "free" of one tile. NPC creeps (keepers/invaders, engine users '2'/'3') never auto-exit ŌåÆ `CombatWorld.npc_owners`. Fatigue zeroes on entering an edge tile (`is_edge`).
- **Combat (per-room):** tower fire only reaches creeps in the **tower's room** (range via world-coords is naturally huge across rooms ŌåÆ out of range); `rampart_at`/redirect become room-aware. Most range checks already use `get_range_to` (room-aware) so this is mostly fixing the `(x,y)` helpers.
- **Agent (`screeps-combat-agent`):** N-room `ScenarioBuilder` (multiple rooms, per-room terrain/structures, exits); **cross-room direction production** ŌĆö the harness must pathfind a creep toward a target in another room and emit the per-tick `Direction` (the engine doesn't pathfind). **Objective beds:** core + towers + ramparts + defender creeps with active rampart-repair + tower-heal-of-defenders (the D5 fight model), with a win condition.
- **Eval (`screeps-combat-eval`):** metrics/cohesion room-aware; extend the tournament to attacker-vs-objective + composition space (ADR 0022 P-AUCTION).

## Build slices (each independently testable)

- **S1 ŌĆö multi-room terrain + edge-exit movement** (foundational substrate): **DONE 2026-06-24** ŌĆö `terrain` default + `rooms` override map + `terrain_for`/`terrain_mut`; `resolve_moves` + the fatigue loop read the destination room's terrain. **Cross-room movement model corrected 2026-06-24** after reading the real engine (`move.js`/`movement.js`/`creeps/tick.js`/`global.js`): the first cut modelled crossing as a `checked_add` *step* (the RTS model); the engine instead **rejects off-edge moves** and crosses via an **automatic edge-exit relocation** of any non-NPC creep on an exit tile (same-tick, NPC-exempt). `step()` reverted to in-room-only; edge-exit relocation added to Phase D; `CombatWorld.npc_owners` added. Tests: `step` rejects off-edge; `resolve_moves` does not carry a creep off an edge; the wall check reads the creep's own (non-default) room; edge-exit relocates an idle creep / is same-tick on move-onto-edge / skips NPCs / a move-inward escapes the exit. Engine 49 tests green, clippy-clean, dependents unaffected (`..Default::default()`).
- **S2 ŌĆö per-room combat:** **DONE 2026-06-24** ŌĆö rampart shield/redirect keyed by full `Position` (`rampart_tiles`/`rampart_id_at`/`on_rampart`/`redirect` in resolve.rs); movement contention keyed by `Position` (`Mover.current_pos`/`dest_pos` + all `want_count`/`matrix`/`want_idx`/`creep_at`/swap/chain-block maps); the dead `(x,y)`-keyed `CombatWorld::rampart_at` removed. A 5-agent adversarial audit confirmed every *other* combat site is already room-safe via `get_range_to` (tower fire is naturally out-of-range across rooms ŌĆö no explicit room-scoping needed; damage/heal pools are id-keyed; Phase-D edge/fatigue is room-local). 2 regression tests: a rampart only shields within its own room; two creeps at the same `(x,y)` in different rooms don't contend. Engine 45 tests green, clippy-clean.
- **S3 ŌĆö N-room ScenarioBuilder + cross-room direction production** (agent): **DONE 2026-06-24.** (a) `ScenarioBuilder::in_room(room)` switches the current room so terrain/structures/towers can be placed per-room; terrain methods now write to that room's override (`terrain_mut`). (b) `CombatCostSource` made **room-scoped** ŌĆö only obstacles *in* the queried room populate its matrix, and terrain reads `terrain_for(room)` (before, every room's obstacles overlaid at the same `(x,y)`, so the search could never route across a border). **Finding: the rover's headless `LocalPathfinder` is intentionally single-room** (cross-room travel is a separate MoveToRoom phase, like the live bot's `find_route` + move-to-exit; `find_route` is a stub there). So cross-room `MoveTo` is handled by **`in_room_goal`** ŌĆö project the cross-room target onto the current room's edge (the exit tile toward the target's room), route to it single-room, and let the engine's edge-exit carry the creep across; the next room re-projects. Tests: `aims_toward_the_exit_for_a_cross_room_target`; `paths_a_creep_across_a_room_boundary` (end-to-end loop of `resolve_move_direction` + `resolve_tick` ŌĆö the creep crosses and arrives). Agent 37 tests green, clippy-clean. **Follow-ups (noted, not S3):** the squad **anchor** mover (rover `AnchorPath`) uses the same single-room `search`, so managed-squad cross-room needs the same MoveToRoom treatment (S4/S5); diagonal room routes clamp toward a corner (cross incrementally ŌĆö refine if a scenario needs it); cross-room `Flee` stays single-room.
- **S4 ŌĆö objective beds with active repair** (the D5 fight model): **DONE 2026-06-24** (`objective_bed.rs` in combat-agent). `defense_intents` mirrors the engine stronghold AI (`stronghold.js`): **focusClosest** (all energized towers + in-range defenders hit the hostile closest to the core) + **towersMaintenance** (one spare tower heals the most-damaged defender, else repairs the most-damaged rampart ŌĆö the active repair). The engine has **no creep-repair action**, so active repair is tower-based ŌĆö which is exactly how strongholds hold (defenders don't repair; towers heal them and static high-HP ramparts wall the core). `run_siege` runs an attacker (any `FnMut(&CombatWorld) -> Intents` ŌĆö `dismantler_intents` for tests, a real attacker AI for the auction) vs the defended core with a win condition: `CoreBreached` / `AttackersWiped` / `Held` (timeout). The bed is composed with the existing `ScenarioBuilder` (S3). Tests: an under-gunned attacker is repelled (repair out-holds its DPS); a TOUGH-buffered sufficient-DPS dismantler breaches; the defense heals a damaged defender while focusing the attacker. **Engine-mechanics finding:** body parts are destroyed **from index 0** (front), so a front-loaded WORK body loses dismantle power to tower fire ŌĆö TOUGH-first is mandatory for a sieger (the test caught this). Agent 40 tests green, clippy-clean. **Follow-ups (noted):** a dedicated dismantle-immune `Core` StructureKind (so the sim requires *attack* not dismantle, per the operator's invader-core rule); defenders that reposition (`assignDefenders`); `focusMax` for L4ŌĆō5; the auction (P-AUCTION) plugs a real attacker composition into `run_siege`.
- **S5 ŌĆö scenarios + integration gate:** **DONE 2026-06-24** (`scenarios.rs` in combat-agent ŌĆö the offline whole-stack gate, ADR 0022 PROVE-1's offline half; run via `cargo test ŌĆ” scenarios`). Each scenario reuses the real `resolve_move_direction` + `resolve_tick` + `run_siege` (not mocks): **CROSS-ROOM-TRAVEL** (a creep routes across *two* borders to a target ŌĆö S3 in_room_goal + S1 edge-exit, re-projecting each room); **FLEE-ACROSS-ROOMS** (a creep flees a far threat to the room edge and the edge-exit carries it across ŌĆö flee + edge-exit, no special cross-room flee); **ATTACKER-VS-OBJECTIVE** (a sieger crosses the border, breaks the rampart, and damages the core under tower fire/repair ŌĆö the full S1ŌåÆS4 stack the auction will score). Agent 44 tests green, clippy-clean. **Findings:** (1) the S1 edge-exit means a target reachable *only* from an exit tile is un-besiegeable ŌĆö the attacker is auto-exited off it every tick; a bed must sit Ōēź1 tile inside the border (the attacker naturally moves off the exit because the target is then range > 1). (2) A *lone* dismantler is under-gunned for a towered core (the tower wears it down during the open core phase) ŌĆö force adequacy is the auction's job (P-FORCE / P-AUCTION), so the gate asserts the stack *fired* (crossed + breached rampart + damaged core), not a win. **Deferred:** `GROUP-UP-THEN-ENGAGE-ACROSS-BORDER` + `STUCK-MEMBER-TIMEOUT` need the squad **anchor** to route cross-room (the rover `AnchorPath` search is single-room ŌĆö the S3 follow-up); they land here once the anchor gets the `in_room_goal` MoveToRoom treatment.
