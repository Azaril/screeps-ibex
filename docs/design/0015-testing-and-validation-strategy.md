# ADR 0015 — Testing & Validation Strategy

- **Status:** Proposed
- **Date:** 2026-06-09
- **Deciders:** William Archbell
- **Related:** **IBEX-023** (zero automated tests); review §9 (test strategy, pre-deploy gates); plan §5 (per-increment validation); ADR [0006](0006-eval-and-iteration-harness.md) (the local-server harness — this ADR **extends** it: 0006 keeps owning the harness, the colony-health score, and the pre-deploy gates; 0015 owns the overall taxonomy and policy and slots 0006's machinery in as its top layers); ADRs [0001](0001-entity-model.md)–[0014](0014-empire-strategy-and-posture.md) (each ADR's validation register maps onto a layer here); companion doc [`../plans/component-test-plans.md`](../plans/component-test-plans.md) (per-component test plans — the *what*, written against this ADR's *how*; includes the gate-deduplication table consolidating overlapping validation promises).

## Context

The corpus has accumulated ~80 distinct validation commitments across the review, the rewrite plan, proposed-fixes, and ADRs 0001–0014 — and **zero tests exist** (confirmed: no `#[test]`, no `#[cfg(test)]`, no `tests/` directory anywhere in the workspace; `screeps-foreman-bench` is the only verification artifact and it asserts nothing). This ADR turns that pile of promises into a small, named set of test layers with explicit assertion rules, speed budgets, and run cadences — and it is the **guidelines** document: per-component plans live in the companion doc and must conform to it.

Three forces shape every decision below:

1. **Build reality.** The workspace's default target is `wasm32-unknown-unknown` (`.cargo/config.toml:5-6`), so plain `cargo test` cross-compiles the test binary to wasm and fails at execution (verified: `cargo test -p screeps-cache` → "not a valid Win32 application"). However — also verified — `cargo test -p screeps-ibex --target x86_64-pc-windows-msvc` **compiles, links, and runs today**: the local screeps-game-api fork's wasm-bindgen externs compile to stubs on host, so the real boundary is *runtime*, not link time. Any test that actually invokes a JS-bound call (`game::rooms()`, `Creep::store()`, `RoomPosition` construction) panics — which on the host dev profile is a clean test failure, because `panic="abort"` is **release-only** (`Cargo.toml:15-16`) and the Cargo test harness always builds with unwind. Logic is host-testable exactly to the extent it sits behind DTO/trait seams that never reach a JS-bound call.
2. **Emergence.** Behavior arises from ~60 systems composing over thousands of ticks, on a server where ticks are concurrent across players. Exact-outcome assertions on emergent behavior are brittle *by construction* — they fail on harmless drift and pass on real bugs. The honest assertion forms for emergent layers are **invariants** ("no creep fires two intents in one tick", "the campaign reaches a terminator") and **distributions** ("median RCL2-by-tick across N seeds ≤ threshold"). Exactness is reserved for the layers where the code under test is deterministic and fully owned.
3. **The operator's two hard requirements.** (a) **High confidence at the composition seams between systems** — the review's real failures (transfer↔jobs accounting, squad↔mission lifecycle, serialize↔everything) live at seams, not inside kernels. (b) **No drag on fast iteration** — strategy and tuning code must remain editable in seconds, not minutes; any test that taxes a weight tweak is a defect of *this ADR*.

Constraint notes that recur below: CPU = execution **+ intents** (see [`../references/engine-mechanics.md`](../references/engine-mechanics.md)) — so host wall-clock benches are regression detectors, never budget validators; the VM can reset any tick — so recovery is a first-class test subject; single-threaded WASM — no test may assume parallel dispatch.

## Decision

Seven named layers; a **seam contract registry** as the composition-confidence centerpiece; an explicit fast-iteration policy with a per-change test tax table and a deliberately-untested list; and concrete tooling picks grounded in the verified build reality. All testing work is **None-breaking**.

### 1. The test taxonomy — seven layers, hard assertion rules

**The assertion rule, stated once:** *exact* assertions (equality, byte-diffs, snapshots) are permitted only at **L0–L3**, where the code under test is deterministic and the inputs are fully owned. **L4** asserts hard invariants and zero-class event counters. **L5–L6** assert *only* invariants and distributions — an L5 test that asserts "X happens at tick 1500 exactly" is rejected at review, full stop. Waits are condition-polls within a tick window, never fixed-tick checks (the Riot rule: no sleeps, no exact-tick asserts).

Three clarifications the layer table implies but the corpus keeps blurring:

- **L5 gates are owned by 0006 and not re-owned here.** This ADR contributes only the *assertion-form rule* (distributional/invariant, N-seed quantile) and the flake policy (§3); scenario definitions, the score, the server machinery, and the pre-deploy gate list stay in 0006 and the companion doc.
- **One artifact per gate.** Where multiple docs promise "the same" gate (the sweep found 17 overlap clusters — e.g. four docs each promising the forced-reset cost-matrix check, three promising the HaulJob parity recorder), the rule is: **one harness artifact, one owner, multiple claimants attach assertions to it.** The companion doc carries the consolidation table; new ADRs must reference an existing artifact before minting a new one.
- **L4's strength is staged, not immediate.** The host lane composes only what is host-constructible, and at Inc 0–1 that is a thin converted subset (serializer + arbiter double, the matcher after its DTO swap, rover, governor/spawn kernels). The "per-change seam confidence" claim reaches full strength only as the class-L refactors land — `CreepState` at Inc 6, the `RoomData`/GameView split across Inc 4–6. Until then L4 confidence is explicitly partial, the L5 gates carry correspondingly more weight, and the companion doc's roadmap table states what L4 can actually compose per increment.

### 2. The composition-confidence mechanism — the Seam Contract Registry

This is the centerpiece, because it is the operator's stated value target. Whole-bot e2e cannot give per-change confidence at seams (too slow, too emergent); unit tests of either side alone cannot either (each side passes against its own assumptions). The mechanism is **in-process consumer-driven contracts**: for every architectural seam, **one contract module** is the single shared source of truth for what crosses the seam, and *both* sides test against it.

**The pattern (uniform across all seams):**

Each registered seam gets one file, `contracts/contract_<seam>.rs`, living in a shared dev-dependency crate **`screeps-testkit`** (new workspace member; host-only; never compiled into the wasm artifact). The file contains exactly four things:

1. **The contract surface** — a re-export or type alias of the seam's trait/DTO (the seam definition itself stays in production code; the contract file *names* it and freezes the invariant list as doc-tested prose + assertion helpers).
2. **Shared fixture builders** — canonical valid/degenerate/adversarial instances of the seam's DTOs (e.g. the market T1–T6 fixtures, a recorded-snapshot loader, a `RoomData` stub builder).
3. **Producer-side contract tests** — the real producer is run and its output is asserted against the contract's invariants (e.g. "`serialize_world` output always begins with a version header"; "every `Plan` the planner emits places structures only on plannable terrain").
4. **Consumer-side contract tests** — each consumer's expectations are run against the shared fixtures, *not* against the live producer (e.g. "the hauler matcher, fed any contract-valid snapshot, never over-commits capacity"; "the scheduler, fed tier=Critical, never invokes a sheddable system").

**The one-shot limit (stated, not implied).** An in-process contract asserts properties of a single call against owned fixtures. The field failures motivating this ADR — the lifecycle hang, commitment churn, renew-as-glue — are **cross-tick temporal failures** that no one-shot contract can express; contracts buy seam-*shape* confidence, not temporal confidence. Every registered seam must therefore also name its **temporal invariants** ("every objective reaches a terminator"; "a committed delivery completes or releases its ticket within N ticks") and the **L4 fault-injection-sim scenario or L5 scenario that asserts them**, in the contract file's header block. A seam whose temporal section is empty is not fully registered.

**The contract tax (the one non-negotiable process rule):** a change to a seam's signature or semantics is allowed **only in the same commit as the contract-pair update**. This is deliberately the *only* mandatory test coupling in the whole strategy — it is cheap (the file is small and colocated), and it is precisely targeted at where the review found the real failures.

**Draft until frozen (when the tax applies).** During the seam-forming increments (Inc 1–6) signatures churn *by design*, and a same-commit rule there would tax exactly the iteration this ADR must not. A registered seam is therefore **draft** from the moment its contract file lands until the owning increment ships and declares the seam **frozen** (the rewrite plan's freeze declarations — e.g. the matcher seam freezes at Inc 1). While draft, contract updates are advisory — encouraged, never merge-blocking. The same-commit tax binds only post-freeze.

**The registry.** Seams are registered here as their ADRs land; the initial registry is the sixteen seams the rewrite defines (S13–S16 added after adversarial review found them missing — the Job trait above all, since it is the FSM rewrite's central seam). "Lands" = the increment in which the contract file must exist (definition of done for the seam itself).

**Debug-build runtime invariants (the contract suite's live counterpart).** Cross-system corruption that no fixture anticipates is caught by **invariant validators** compiled into debug builds: ECS referential integrity, no-double-intent per (creep, category) per tick (IBEX-029/ADR 0003 §A), transfer capacity conservation (IBEX-045), `is_finite` on every float priority (IBEX-046), task/mission-graph acyclicity. The `panic="abort"` interaction is resolved by one macro with two behaviors:

- **`ibex_invariant!(cond, "msg")`** expands to `debug_assert!` on non-wasm builds (host tests and the L4 sim run them hot — a violation is an immediate, attributable test failure; the test harness always unwinds, per the Cargo profile rules `Cargo.toml:15-16` being release-only), and to **log-and-count** (increment a seg-57 counter, continue) on `target_arch = "wasm32"` — because in deployed wasm a panic costs the whole tick, and in MMO release `panic="abort"` makes it an instant full abort.
- Build-profile matrix, decided: **host tests + L4 sim: validators on, panicking. Debug wasm deploys to the private server: validators on, logging (behind the existing-style cargo feature, default-on for harness builds). MMO release: validators compiled out entirely.** Never an asserting validator in any wasm artifact.
- **L6 visibility, documented:** because MMO release compiles the validators out, the live-MMO canary (L6) sees only the always-on seg-57 counters — `deser_failures`, `panics_caught`, `restart_counter`, `segment_fullness_watermark`, the death-spiral signal block — and is **blind to the `ibex_invariant!` validator classes** (double-intent, capacity conservation, `is_finite`, acyclicity). Those classes are confirmed only by host tests, the L4 sim, and debug-wasm private-server runs; the MMO stream cannot vouch for them.

### 3. The fast-iteration policy — explicit and budgeted

**The kernel/shell rule (the policy's spine).** Code is classified once, at the module level, in the companion doc:

- **Stable kernels** (the ~25 named pure decision functions: body math, threat classification, formation geometry, governor tier, matcher selection, encode/decode, plan scoring, posture engine, trade governor, …) **require L0 + at least one L2 property or relation**. Kernels are where exactness is honest and where tests are assets that survive refactors.
- **The experimental shell** (mission `run_mission` bodies, utility weights, scoring constants, body-composition tuning, shed-order tuning, market thresholds — everything expected to churn weekly) is pinned **only** by L5 scenario gates plus, where useful, insta snapshots whose entire maintenance cost is one `cargo insta review` keystroke. **The shell is never unit-mocked.** A unit test that mocks four neighbors to pin a strategy choice is test bureaucracy: it breaks on every tweak and catches nothing the scenario gate wouldn't. This rule is enforceable at review: "is this module shell? then why does this PR add a mock?"

**The per-change test tax table.** What a change costs, by class — and nothing more:

| Change class | Required test work | Required runs before merge |
|---|---|---|
| Bugfix in a kernel | Failing L0 test reproducing the bug, first | L0–L2(fast) + contracts |
| New kernel | L0 cases + ≥1 L2 property/relation | same |
| Strategy/tuning tweak (shell) | **None.** Re-baseline any diffed insta snapshot (one keystroke); nightly L5 covers behavior | L0–L2(fast) + contracts (they should be untouched) |
| New mission / strategy feature (an **existing** scenario covers the behavior) | A scenario assertion added to that existing 0006 scenario (invariant/distributional form) — cheap, required; no unit tests | same + the assertion in the next nightly |
| New mission / strategy feature (needs a **new** scenario) | The feature merges behind the existing gates; authoring the new scenario is a **separately budgeted harness task** (an F9 row in the companion doc), **not a merge precondition** — it lands report-only and earns gate status per the flake policy below | same; the scenario enters the nightly when it lands |
| Seam signature/semantics change | **Frozen seam: contract-pair update in the same commit** (§2). Draft seam (pre-freeze): advisory only — update when convenient (§2). If it is a cutover seam, L3 parity green before cutover | + L3 |
| Serialization format touch | S1 contract + round-trip + corpus + fuzz green (the ADR 0002 hard gate) | + nightly fuzz before deploy |
| Harness/CI/testkit change | Smoke loop green | L0–L2 + contracts |

**Suite speed budgets (enforced — a layer that busts its budget gets tests demoted to merge/nightly, not a bigger budget):**

- Per-change lane (L0 + L1 + L2-without-fuzz + L4 contract suite): one `cargo test-host` invocation, **< 2 min wall-clock including incremental compile** — this is the single per-change law (it resolves an earlier draft's conflicting "< 60 s lane / < 2 min contracts" pair). The layer table's per-layer figures are execution-time guidelines inside it, with pure test execution targeted **< 60 s**; cold/full rebuilds are exempt, and CI measures the law on a warm cache.
- L3 parity suite: **< 5 min**, merge-to-master lane.
- L4 fault-injection sim: **< 5 min**, merge lane.
- Nightly: fuzz (10 min/target), full L5 scenario suite (0006's budget), criterion compare; weekly: cargo-mutants audit.
- Pre-deploy: 0006's gates, unchanged.

**The flake policy.**

- **L0–L4 are deterministic by construction** (seeded RNG everywhere, no wall-clock, no network). A flaky test at these layers is a *bug in the test or a determinism bug in the code* — both are real defects; fix or delete within a day. **CI retries are forbidden at L0–L4** (a retry config is how flakes become permanent).
- **L5 is distributional by construction:** every scenario gate runs **N seeded runs** — N per gate class lives in the companion doc's F19 thresholds file (the single source; current defaults there: nightly N=5, increment/pre-deploy gates N=9, raised per-gate where the failure class warrants it) — and gates on the **paired-seed diff against the stored (scenario, seed, SHA) baseline**, which the harness already records (F14). The standing form is "**≥k of N matched seeds within tolerance of the same seed's baseline, and the median diff ≤ threshold**" — not an absolute threshold: a small-N absolute gate of the "median ≤ T and no seed > 2T" shape misses regressions that hit under half the seeds, and an absolute value "pinned when the gate first runs" is tautologically green at birth. Absolute thresholds are reserved for physical bounds (zero panics, engine deadlines). Seeds are recorded in the run artifact; any failure is reproducible by (scenario, seed, git SHA).
- **Auto-quarantine (the Riot earned-promotion rule, adapted — nightly lane only):** an L5 gate that flips pass↔fail across ≥2 of the last 10 nightlies *without a relevant code change* is auto-demoted to report-only and a tracking issue filed; it is promoted back to gating only after **7 consecutive green nightlies**. New scenario gates *start* in report-only and earn gating status the same way. Gates are earned, not declared — **on the nightly lane**. **Increment-gate mode is the explicit exception** (without it, earned gating would deadlock the rewrite plan's increment gates, which *are* these scenarios): a scenario named as an increment gate gates **immediately and on demand**, as N=9 seeded runs (F19) evaluated as paired-seed diffs against the increment's start baseline. The same scenario can gate an increment while still earning its nightly status.
- The determinism prerequisite gets an owner here: **decision code must iterate deterministically** (no `HashMap`/`HashSet` iteration order reaching decisions — the known offender is spawnsystem.rs:284-289; use `BTreeMap` or sort at the boundary). This is required for L3 byte-diffs and L5 seed-reproducibility alike; it is checked by the L3 suite itself (a nondeterministic system fails parity against *its own recording*).

**Deliberately untested (enumerated, because silence is how test bureaucracy creeps in).** No test will be written for, and no reviewer may demand tests for:

- Mission/operation `run_mission` bodies at unit level (emergent shell — L5 only).
- Exact pathfinding routes or room-placement aesthetics (snapshot the *plan*, not the path; assert route *properties* at L2 if needed).
- Utility weights, scoring constants, shed-order tuning values (L5 + score regression own these).
- Visual rendering beyond payload-shape snapshots (`VisualBackend` collector); per-tick visual call counts.
- MMO CPU exactness (private-server CPU ≠ MMO; criterion is a regression proxy only — engine intent costs per [`engine-mechanics.md`](../references/engine-mechanics.md) are validated by the seg-57 stream, not by tests).
- The screeps-game-api fork's internals (upstream CI covers it; our boundary is the DTO seam).
- The 8-hour power-creep respawn cooldown and other untick-compressible mechanics (kernel-test the policy, never a faithful scenario — ADR 0013's caveat, generalized).
- Pixels / MMO-only features (gap G-15 — marked untestable-by-harness).
- Deploy scripts and js_tools (the smoke loop *is* their test).
- Coverage percentages (see §4 — coverage is informational, never a gate).

### 4. Tooling decisions

| Concern | Decision | Grounding |
|---|---|---|

### 5. Patterns adopted / rejected

| Pattern | Verdict | Why, for *this* system |
|---|---|---|
| Replay intent-diff parity (record/replay + byte-diff) | **Adopt** (L3) | The explicit cutover-gate form for Inc 1 and Inc 6 — at Inc 1 in its **shadow-dispatch** form (same live tick, diffed intent sinks; §Migration step 2), with record/replay proper landing once GameView is real; the only exact-assertion form that survives 60-system composition, because it diffs *our own* deterministic output against itself. **Necessary, not sufficient:** parity is bounded by its recordings (≥3 pinned, including a pressure run) — off-recording behavior belongs to the L5 gates |
| Consumer-driven contracts (in-process) | **Adopt** (L4, §2) | Literally the stated value target — seam confidence per-change at unit-test cost; replaces brittle whole-bot e2e |
| Golden-master / approval (insta) | **Adopt** (L2) | Pins the churning shell at one-keystroke maintenance cost — the only pin compatible with the no-iteration-drag requirement |
| Metamorphic relations | **Adopt** for planner/economy/threat kernels (L2) | The honest oracle when expected output is unknowable: *more energy in never fewer spawns; an added obstacle never shortens a path; a raised CPU budget never completes fewer tasks; an extra hostile never lowers the threat score; raising a transfer priority never demotes it*. Relation design is budgeted domain work in the companion doc, not free |
| Seeded fault-injection sim (VOPR-style) | **Adopt** (L4 host sim; L5 server-side per 0006) | Directly targets the three observed failure classes — lifecycle hang, CPU death-spiral, dangling refs — with seed-reproducible failures; asserts *recovery invariants*, not outcomes |
| Statistical N-run quantile gates | **Adopt** (L5 only) | The honest assertion form for emergent behavior; single-run exact gates are flake generators by construction |
| State-hash / CRC drift check (Factorio) | **Adopt, narrow form** | A post-run hash of serialized world state appended to L4 sim scenarios catches cross-system drift cheaply; *not* applied at L5 (concurrent-tick servers aren't hash-stable) |
| Runtime invariant validators | **Adopt** (debug builds only, §2) | Turns emergent corruption into immediate attributable failures in host sim; the `ibex_invariant!` macro resolves the `panic="abort"` hazard |
| Mutation testing (cargo-mutants) | **Adopt, periodic audit only** | Good test-quality instrument for kernels; per-commit cost is incompatible with the 60-second budget |
| screeps-server-mockup | **Reject** | Stale (last release 2020), JS-centric, pins a divergent server version; 0006 already owns a real-server path with the same one-tick-at-a-time control |
| wasm-executed tests (wasm-bindgen-test runner) | **Reject** | Verified unnecessary: host link works against the fork; the boundary is runtime JS calls, which the DTO rule keeps out of test scope anyway |
| Coverage threshold gates | **Reject** | Manufactures shell tests; violates the iteration requirement. Coverage stays informational |
| Whole-bot e2e per-change | **Reject** | Economics: orders-of-magnitude slower than the kernel lane; its job is done better by contracts (per-change) + scenarios (nightly) |
| TDD / test-first mandate on shell code | **Reject** | The shell is supposed to churn; pin it with snapshots and scenarios, not discipline |

## Alternatives Considered

| Option | Pros | Cons |
|---|---|---|
| **Layered taxonomy + seam contracts + distributional eval (chosen)** | exact tests only where exactness is honest; per-change lane < 60 s; confidence concentrated at the seams where the review found the failures; every expensive layer is nightly with reproducing seeds | upfront testkit/contract investment; two test lanes (host + wasm build-check) to keep green; relies on 0006 landing for L5 |
| Conventional pyramid (unit-heavy, mock the neighbors) | familiar; high unit counts look rigorous | mocks at seams test assumptions, not contracts; unit tests on emergent shell break on every tweak — directly violates the iteration requirement |
| Server-integration-heavy (test everything on the private server) | maximum fidelity | minutes-per-run kills per-change feedback; exact assertions flake on emergence; the community's own consensus is to prefer host tests by orders of magnitude |
| Minimal testing (ship + watch, the Overmind posture) | zero drag | demonstrably how Ibex got here: zero tests, five field-report failure classes, one extinction-class bug |
| In-process mocked server (screeps-server-mockup) | no Docker; deterministic-ish | stale, version-divergent, JS toolchain; duplicates 0006's real-server investment |

## Consequences

**Positive.**
- The operator's two requirements become structural: seam confidence is a per-change, sub-2-minute contract suite; iteration drag is bounded by an explicit tax table whose strategy-tweak row is "none".
- Every validation commitment in the corpus now has a layer, an assertion form, and a cadence; the 17 duplicate-gate clusters collapse to single owned artifacts (companion doc).
- The verified host-test path means the first **~60** kernel tests (tranche 1) need **zero production refactors** — IBEX-023 starts closing in days; the next ~40 unlock behind the two mechanical tranche-2 DTO swaps.
- Failures at every layer are seed- or input-reproducible: L0–L4 deterministic, L5 by (scenario, seed, SHA).

**Negative / costs.**
- The contract tax is a real (small) per-seam-change cost, and `screeps-testkit` is a new crate to maintain.
- Two build targets must stay green forever (the dual-target check makes this cheap but permanent).
- Insta re-baselining can be rubber-stamped; mitigation is review attention on snapshot diffs, which is process, not tooling.
- L5 confidence arrives only as 0006's scenario library fills in — until then the top of the pyramid is thin and the bot leans on L0–L4 plus MMO canary.

**New risks.**
- The `GameView` trait (S5) is load-bearing for both L3 and the DTO rule but has no owning ADR yet; this registry row forces the question but does not answer it (flagged to 0001/0005).
- Determinism debt (HashMap iteration) will surface as L3 failures; that is intended, but the first parity runs will be noisy until the named offenders are fixed.
- Fixture realism drift: contract fixtures that no longer resemble live data give false confidence; mitigated by sourcing fixtures from recorded snapshots wherever a recorder exists (Inc 1+).

## Incremental Migration Path

All steps **None-breaking** (test-only code, host-only crates, mechanical DTO swaps with no behavior change; labels per step regardless).

1. **Inc 0 — stand up L0–L2 + the host lane + the first ~60 (then ~100) kernel tests.** *(Breaking: None.)* Add the `test-host` cargo alias + CI dual-target check; create `screeps-testkit` (fakes: FakePathfinder, FakeCreepHandle, MemoryArbiter-double, fixture builders); add proptest + insta + cargo-fuzz scaffolding; write tranche 1 (~60 tests, zero production edits: serialize round-trip/corruption, `create_body`, damage falloff, `classify_threat`, repairqueue ordering, spawn comparator, stats cascade, machine_tick cap, findnearest combinators, squad geometry, rover resolver, foreman layer snapshots vs the bench corpus); land the two tranche-2 DTO swaps (`RoomPosition`→`Position` in transfersystem ranking; body-DTO param for `analyze_hostile_creep`) and write the unlocked transfer/threat tests (~40 more). Register the first contracts: **S1, S2, S12** (+ S3's rover half). First fuzz target on decode. Wire `screeps-foreman-bench` into the workspace as the pre-deploy plan-quality gate **now** (host-only, cheap — ADR 0009 step 2; aligned with the companion doc's F15 Inc-0 placement). This rides Inc 0 alongside 0006's smoke loop and satisfies plan §3's "first pure-logic kernel tests" deliverable.
2. **Inc 1 — scheduler shadow-dispatch parity + the intent differ + governor and facade contracts.** *(None.)* The Inc-1 parity gate does **not** ride record/replay — at Inc 0–1 `GameView` is a skeleton (legacy systems still call JS directly), so a recorder riding it would record a thin slice (false confidence), while a real recorder blocks on the class-L refactors. The Inc-1 mechanism is **in-process shadow-dispatch**: run the old and new schedulers over the *same live tick*, the new side writing to a shadow intent sink that is never executed, and **diff the two intent sinks** tick-by-tick — no replay, no recording, no GameView dependency; the determinism prerequisite shrinks to ordering-stability of the diffed sinks (sort at the sink). Build the **intent differ** now — it is the differ half of the F4 artifact, reused verbatim by every later parity gate; the **recorder proper** is deferred to the increment where GameView is real (step 5), and the reference recordings are captured then. Earlier cutovers that need parity before the recorder exists (squad store at Inc 3, orchestrator at Inc 4) use the same shadow-dispatch mechanism. Register **S3, S4, S5**. Stand up the L4 fault-injection sim (seeded resets, corrupted segments, squeezed budgets; recovery invariants + state-hash check). `ibex_invariant!` macro + first validators (double-intent, is_finite, capacity conservation).
3. **Inc 2 — serialization hard gate.** *(None — the format change itself is 0002's labelled reset.)* MemoryArbiter double in testkit; begin the **old-snapshot corpus** (capture convention: every deployed schema version contributes payloads under `fuzz/corpus/` + `tests/snapshots/`); fuzz suite becomes the S1 hard gate per ADR 0002.
4. **Inc 3 — S8 (SquadStore) contract + identity fixtures** (round-trip, recycled-slot, stale-ref worlds). *(None.)*
5. **Inc 4–5 — S7, S9, S11, S14 contracts** as the spawn orchestrator, Squad Manager, posture resource, and `SquadContext`/`new_with_squad` land; the **F4 recorder proper** lands here with the GameView-realization work (hard-required before the Inc-6 pilot) and the ≥3 reference recordings (economy-bringup, engagement, induced-pressure) are captured; cohesion + teardown-deadline scenarios enter the 0006 library (report-only on the nightly lane; increment-gate mode applies per §3). *(None.)*
6. **Inc 6 — FSM pilot reuses the recorder** (landed with the GameView work, step 5) **and the Inc-1 differ** for HaulJob byte-parity; job-FSM states become L0-testable via `machine_tick` replay as the `CreepState` DTO refactor lands; register **S13** (the Job trait — the FSM rewrite's central seam) with it. *(None.)*
7. **Inc 7+ — S10 (market) + S15 (RoomGraph) contracts (S16 at Inc 8); S6 extensions.** The S6 matcher contract itself **lands at Inc 1** (registry row — an earlier draft said "full matcher at Inc 7+", an inconsistency resolved in favor of Inc 1 to match the rewrite plan's frozen matcher seam); Inc 7 *extends* the existing file with the route-distance/multi-room sizing clauses, it does not create it. Weekly cargo-mutants audits begin once kernel crates stabilize. *(None.)*

**Standing rule from here on:** a new seam lands **with** its contract file (definition of done; draft until its increment freezes it, §2), and a new scenario gate starts report-only on the nightly lane and earns gating status (increment-gate mode excepted, §3).
