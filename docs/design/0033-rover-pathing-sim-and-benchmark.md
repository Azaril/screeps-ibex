## Context

### ¬ß0 The pain, and why now

`screeps-rover` is the single most-leaned-on subsystem in the bot that has **no first-class validation of its own quality or cost.** Every hauler, upgrader, claimer, scout, and combat squad routes through it. The combat program now depends on it heavily ‚Äî multi-room A* (`LocalPathfinder::search`), the traffic resolver wired in via `resolve_moves_via_system`, anchor footprint pathing (`AnchorPath`), the fatigue gate. Yet:

- The combat sim (ADR 0023) **runs rover but never measures it.** Since P-MOVE+ (task #30) the sim drives the *live* rover stack ‚Äî `MovementSystem` + resolver + `LocalPathfinder` + `AnchorPath` ‚Äî to produce each tick's move directions (`resolve_moves_via_system`, `pathing.rs:477`; the single-creep `resolve_move_direction`, `pathing.rs:198`), then hands them to the combat-engine's `resolve_tick` and scores the *fight* (DPS, healing, cohesion, survival). Rover is **exercised on every tick, measured on none** ‚Äî no combat metric grades its path optimality, fatigue/time-moving, congestion, ops, or determinism. (0023's older "Architecture finding" line ‚Äî "it does NOT run the bot's rover/`MovementSystem`/`AnchorPath` ‚Ä¶ validated by rover unit tests + live, **not** here" (`0023:11`) ‚Äî predates that wiring and is **stale**; 0023's own P-MOVE+ status line is the current truth. This is exactly the "trust the code, not the combat-status docs" hazard this project has hit before.)
- Seam S3's gates (ADR 0004 / `0015:71`) are real but **thin** ‚Äî "ops never exceed the per-tick pool; `MIN_PATHFIND_OPS` floor at Critical; unreachable ‚Üí `Failed`, never a hang." There is no harness that drives them richly over a corpus.
- The only existing correctness evidence is **ad-hoc unit tests** in `resolver.rs`/`local_pathfinder.rs` plus the determinism fence that was discovered **accidentally** ‚Äî a kiting combat test flaked ~50% of the time and the root cause turned out to be three seed-ordered `std::HashMap` iterations in `resolver.rs` (`topological_sort_follows`, `current_pos_to_entity`, swap discovery), fixed with `Handle`-sorted tie-breaks. A mover regression today silently corrupts every squad and every economy hauler, and we would learn about it the way we learned about the seed-flake: by something downstream breaking.

The operator's headline asks are concrete: *"time creeps spend moving, efficient usage of fatigue, optimal pathing"* and *"algorithmic/cpu benchmarking."* Neither is measurable today. This ADR designs the harness that makes them numbers.

### ¬ß1 The constraint litany (the forces every ADR in this corpus respects)

- **Single-threaded WASM, per-tick CPU budget including intents.** Movement is a large slice of that budget (`MAX_PATHFIND_OPS=20_000`, `DEFAULT_PATHFIND_OPS_BUDGET=20_000`, the live `MovementSystem` already meters `ops_consumed` and sheds load). The benchmark's primary CPU number must therefore be **op count** (deterministic, MMO-relevant), with host wall-clock as a *secondary* regression proxy only ‚Äî host wall-clock ‚âÝ MMO CPU and is blind to intents (`0015:20`, `0015:133`).
- **VM-reset resilience** ‚Äî irrelevant to an offline host harness directly, but the persisted per-creep `StuckState`/`CreepPathData` cache that the sim must own across ticks mirrors the across-tick state the live bot persists.
- **Incremental migration with a stable seam.** The seam already exists: the rover crate's `screeps` cargo feature. WITHOUT it the crate compiles and runs on native targets using only pure data types from `screeps-game-api` (`Position`, `RoomName`, `LocalCostMatrix`, `Direction`) ‚Äî no JS interop. WITH it, `screeps_impl.rs` provides the real-game trait implementations. An offline simulator is just a second consumer of that already-shipped seam.
- **Determinism is doctrine.** The project's determinism fence (`sim_is_deterministic_over_rounds`, spread 0) is non-negotiable, and `resolver.rs` is the *documented historical source* of nondeterminism. A rover benchmark is exactly where a directly-targeted rover determinism gate belongs.

### ¬ß2 The one architectural fact that frames the whole design: rover plans, it does not simulate physics

`MovementSystem::process()` (`movementsystem.rs:459-797`) turns intents into a per-creep `Direction` (or a pull pairing) and **stops**. It *reads* `creep.fatigue()` and `creep.spawning()` to gate a creep (`:510-514` ‚Äî `if fatigue() > 0 || spawning()` ‚Üí report `Moving`, skip), and it *writes* the move via `move_direction(dir)`. It never:

- computes fatigue accrual or decay,
- resolves the engine's authoritative tile-contention winner (engine `rate1`/pulled/pulling/moves-weight ‚Äî `engine-mechanics.md ¬ß1.6`; rover's *own* resolver uses a different RTS tie-break: `(priority, stuck_ticks, Handle)`, `resolver.rs:424-443`),
- relocates a creep across a room edge, or
- moves a creep at all.

Those are the **server's** job. The clean reading: there are **two distinct movement resolvers** ‚Äî rover's (navigation quality: which tile/path, cheaply, deterministically) and the engine's (the authoritative adjudicator of the moves rover requests, with fatigue and weight contention). This is confirmed by the only existing offline `process()` caller, `screeps-combat-agent/src/pathing.rs`: `CombatCreepHandle::move_direction` records the direction into a sink (`:333-336`); `fatigue()` returns a value the *engine* set (`:327-329`); `resolve_moves_via_system` (`:477-518`) runs rover, collects directions, and hands them to the combat-engine's `resolve_tick`, which alone applies movement + fatigue (`resolve.rs:494-518`) and resolves real contention (`movement.rs`).

**Consequence for this ADR:** the rover sim is a **two-halves tick loop** ‚Äî (A) rover `process()` = the planner under test; (B) the authoritative "server" half ‚Äî the combat engine's movement tick, **reused** (extracted to `sim-core` as `resolve_movement`, ¬ßD1/¬ßD3), not re-built ‚Äî that applies moves, accrues/regenerates fatigue, resolves engine-true contention, and relocates edge-crossers. The benchmark measures the quality of (A)'s decisions against a faithful (B). **A "fatigue efficiency / time-moving" metric therefore lives entirely in half (B); there is nothing to measure inside rover for fatigue** ‚Äî rover's whole fatigue model is the single binary read at `:510`. Conflating the two halves (trusting rover's own resolved `final_pos` as if it were the executed move) would hide the exact failure class the sim exists to surface, because the two resolvers use different tie-break rules.

### ¬ß3 What is in scope vs out of scope

**IN scope** ‚Äî offline, deterministic, host-only validation of rover for:

- single-room and N-room (‚â§ `MAX_PATHFIND_ROOMS`) **pathing** ‚Äî route optimality, length, oscillation, incomplete/fail rate, ops-per-search;
- **movement execution / fatigue** ‚Äî ticks-to-arrive, fatigue stalls/waste, terrain-rate (road 1 / plain 2 / swamp 10), sustained-speed inequality, edge-zeroing, power-creep no-fatigue, the idle decomposition (fatigued vs blocked);
- **multi-creep congestion** ‚Äî shove/swap/local-avoidance, trains/pulls, deadlock/livelock detection, throughput/makespan;
- **flee/kite** correctness and **anchor** stability;
- **algorithmic/CPU** cost ‚Äî ops-count gates (primary) + wall-clock regression (secondary), scaling curves;
- **determinism** as a first-class gate.

**OUT of scope** (owned elsewhere, do not duplicate):

- **combat resolution** ‚Äî damage, healing, towers, focus-fire, kite *decisions*: owned by ADR 0023 / `screeps-combat-engine`. This sim consumes a body and a route; it never fires a ranged attack.
- **economy / colony / the full game tick** ‚Äî owned by ADR 0006's colony-health sim.
- **the S3 seam contract shape** (ops pool semantics, `MIN_PATHFIND_OPS` floor) ‚Äî owned by ADR 0004 / 0015; this sim *exercises and gates* S3 but does not redefine it.
- **exact pathfinding routes as golden snapshots** ‚Äî explicitly forbidden by `0015:130` ("snapshot the *plan*, not the path; assert route *properties* at L2"). This sim asserts route **properties and distributions**, never "the path is exactly `[tiles]`."

This is the **rover-scoped realization of the L0‚ÄìL6 taxonomy**, the movement analogue of ADR 0006's combat micro-sim.

## Decision

**Reuse the combat sim's mover; do not rebuild it.** The combat sim is already a working two-halves mover ‚Äî rover *plans* (`resolve_moves_via_system` runs the real `MovementSystem` + resolver) and the engine *applies* (`resolve_tick` does fatigue / contention / edge-exit) ‚Äî it just scores combat, never the mover (see ¬ß0). So this ADR makes **two moves**: (1) **extract `screeps-sim-core`** ‚Äî the general, combat-agnostic Screeps movement mechanism (world, terrain, body, the same-tile contention resolver, the move/fatigue/edge-exit tick, the recording, the deterministic RNG, and the offline rover driver) lifted out of `screeps-combat-engine` into a shared lower crate **both** sims depend on; (2) **create `screeps-rover-eval`** ‚Äî a host-only *measurement* crate that drives the *real* rover `MovementSystem` / `LocalPathfinder` / `resolver` over a `screeps-sim-core` world, applies the issued moves through sim-core's authoritative movement tick (**no re-ported physics, no new `RoverWorld`**), and emits a gated rover-quality metric set (path optimality, fatigue / time-moving, congestion, ops) with **determinism as the headline ship-blocker**. The only net-new *physics* is two fidelity fixes ‚Äî **roads** and **loaded-CARRY** fatigue ‚Äî that land in `screeps-sim-core` and improve **both** sims.

> **Note (decision provenance):** an independent reuse-surface analysis recommended *against* the extraction ‚Äî it found the shared primitives "already correctly layered" in `screeps-combat-engine`/`screeps-combat-agent` and judged a `screeps-sim-core` crate "churn for zero new sharing today." The operator chose the extraction anyway, for the **cleanest end state** (operator directive: *target the cleanest design, not incremental tech debt*) ‚Äî **clean layering** (a movement bench should not depend on a crate named `combat-engine` or a world named `CombatWorld`), a full `Combat*`‚Üí`Sim*` rename of the shared kernel, and a **reusable mechanism foundation** for future non-combat sims (economy / hauling / lifecycle ‚Äî ADR 0028, `screeps-ibex-eval`). The trade-off (extraction + rename churn vs. lower-churn reuse-in-place) is recorded in [Alternatives](#alternatives-considered); the component-level reuse map below is unchanged by the choice ‚Äî only the *home crate and names* of the shared code differ.

The remainder of this section specifies: the two new crates and the extraction (¬ßD1); the anti-duplication map of what is reused vs net-new (¬ßD2); the reused server half + the two shared physics fixes (¬ßD3); the ground-truth oracle (¬ßD4); the metric set with formulas and L-layer mapping (¬ßD5); the scenario catalog (¬ßD6); the determinism fence (¬ßD7); and where each gate sits in the 0015 taxonomy (¬ßD8).

### ¬ßD1 The two new crates and the extraction

The design adds **one new lower crate** ‚Äî `screeps-sim-core` (the shared mechanism; wasm + host, a submodule like the others) ‚Äî and **one new host-only crate** ‚Äî `screeps-rover-eval` (the measurement layer). There is **no `RoverWorld` and no re-ported physics**: the substrate is the combat sim's, factored out.

#### ¬ßD1a ‚Äî `screeps-sim-core`: the pure movement kernel ‚Äî **AS BUILT (M0 landed 2026-07-01)**

> This section is reconciled to the shipped code. Earlier drafts explored other shapes (`SimWorld` + a separate `CombatState`, a `resolve_combat_tick` rename, no `CombatWorld` type); the as-built decisions below supersede them, and the [Alternatives](#alternatives-considered) records why.

`screeps-combat-engine` bundled two separable concerns: (1) **general Screeps movement mechanics** ‚Äî terrain, body-sizing arithmetic, the same-tile contention resolver, fatigue, edge-exit, the per-tick movement world ‚Äî and (2) **combat** ‚Äî damage, towers, ramparts, controllers, safe-mode, body-combat powers. M0 lifted (1) into `screeps-sim-core`; combat-engine keeps (2) and now depends on the kernel. The kernel carries **no combat concept**.

**What the kernel holds (as built):**

| kernel module | contents |
|---|---|
| `world.rs` | `MovementState { tick, terrain, rooms, creeps, npc_owners }` (the movement world) + `SimCreep` + `CreepId` / `PlayerId` / `StructureId` |
| `terrain.rs` | `SimTerrain { walls, swamps }` + `fatigue_rate` / `is_wall` |
| `body.rs` | `SimBody` ‚Äî **movement / body-sizing arithmetic only** (`move_rate`, `fatigue_weight`, `can_move`, `fatigue_clear`, `part_hits`, `effective_power`, ctors); no combat powers |
| `movement.rs` | the same-tile contention resolver ‚Äî `resolve_moves` / `resolve_moves_with_pulls`, `step`, `is_edge` (moved verbatim from the engine port) |
| `tick.rs` | `resolve_movement(&mut MovementState, &MoveIntents) -> MovementReport` ‚Äî the movement tick: contention ‚Üí apply move/fatigue ‚Üí edge-exit ‚Üí `tick += 1` |
| `intents.rs` | `MoveIntents { moves, pulls, reasons }` ‚Äî the base action vocabulary |
| `sim.rs` | the `Simulation` layering trait + `MovementSim` (the movement-only layer) |
| `constants.rs` | `BODYPART_HITS` + `FATIGUE_*` (movement / body-sizing only) |

**The combat layer (`screeps-combat-engine`), as built** ‚Äî a thin layer on the kernel, **byte-identical** (55 tests preserved: kernel 19 + engine 36):

- **`CombatWorld` = composition, not a rename.** `CombatWorld { movement: MovementState, towers, structures, controllers, safe_mode_owner }`. The name stays (it *is* a combat-layer type); its movement fields now live in the embedded `movement`. **No `Deref`** ‚Äî combat reads `world.movement.creeps` explicitly and the mover takes `&mut MovementState` (combat passes `&mut world.movement`). Chosen over the two rejected alternatives: (a) separate `SimWorld` + `CombatState` params threaded through every helper (~187 edits, ~89 combat-field re-routes, no clarity gain), and (b) `Deref` composition (the operator ruled out the implicit IS-A coupling as shim-like).
- **`resolve_tick` keeps its name**; only its *movement phases* were extracted. It now reads: accumulate damage/heal from tick-start positions (Phase A/B) ‚Üí `sim_core::resolve_movement(&mut world.movement, &moves)` (which owns contention + move/fatigue + edge-exit + `tick += 1`) ‚Üí net damage ‚Üí deaths ‚Üí structures/towers/controllers. **Byte-identical** because per-creep *position* (moved by the kernel) and *hits* (netted here) are independent, edge-exit is occupancy-blind (relocating a soon-dead creep before it is retained changes nothing observable), and `report.tick` is captured before the mover increments it.
- **Body-combat lives in the layer, not the kernel.** The kernel's `SimBody` has only movement / body-sizing methods; the body-combat arithmetic (`attack_power`, `ranged_attack_power`, `heal_power`, `ranged_heal_power`, `dismantle_power`, `damage_after_tough`) + its constants (`ATTACK_POWER` ‚Ä¶ `CREEP_LIFE_TIME`) live in combat-engine as the **`SimBodyCombat` extension trait** (`impl SimBodyCombat for SimBody`), brought into scope where combat calls them. This applies the boundary rule literally: *the mover never needs body-combat, so it isn't in the mover's crate.*
- **Combat's `Intents` keeps its `moves` / `pulls` maps** and `resolve_tick` builds a `MoveIntents` from them to call the mover. (Having `Intents` *embed* a `MoveIntents` is a small clean-up follow-up; the type-level guarantee "a movement sim can't emit a combat action" already holds via `MovementSim::Intents = MoveIntents`.)

**The `Simulation` trait ‚Äî per-layer action-typing (the operator's requirement).** A sim layer is a `(World, Intents, Report)` triple with a tick resolver:
```rust
pub trait Simulation { type World; type Intents; type Report;
    fn step(world: &mut Self::World, intents: &Self::Intents) -> Self::Report; }
```
`MovementSim` implements it with `Intents = MoveIntents`, so a decision routine driving the movement sim **cannot express a combat action at the type level**; a combat / economy layer's `Intents` embeds `MoveIntents` plus its own verbs. A new layer adds its own `(World containing MovementState, Intents embedding MoveIntents, Report)` with **zero** kernel changes ‚Äî the generalization test the layering had to pass. (combat-engine currently exposes the concrete `resolve_tick` rather than an `impl Simulation for CombatSim`; adding that impl is a small follow-up.)

**Naming ‚Äî as built:**

| concern | kernel (`screeps-sim-core`) | combat layer (`screeps-combat-engine`) |
|---|---|---|
| movement world | `MovementState` (ex-`CombatWorld` movement fields) | `CombatWorld { movement: MovementState, ‚Ä¶ }` (composition) |
| terrain | `SimTerrain` (ex-`CombatTerrain`) | ‚Äî |
| creep / body | `SimCreep`, `SimBody` (movement methods) | `SimBodyCombat` extension trait (combat methods) |
| move intents | `MoveIntents` | `Intents` (combat; converts to `MoveIntents` for the mover) |
| tick | `resolve_movement` | `resolve_tick` (calls the mover) |
| layering | `Simulation` trait + `MovementSim` | `CombatSim` impl = follow-up |
| combat entities | ‚Äî | `SimTower` / `SimStructure` / `SimController` / `StructureKind` (unchanged) |

Only genuinely combat-agnostic types were renamed *into* the kernel (`CombatTerrain`‚Üí`SimTerrain`; the `CombatWorld` movement fields ‚Üí `MovementState`). **Combat-layer types kept their names** (`CombatWorld`, `resolve_tick`, `Intents`) ‚Äî the kernel is clean, so there is no reason to churn them (or the bot). `screeps-combat-engine` keeps its name (it *is* the combat tick engine, now layered on the kernel); `screeps-sim-core` reads as the foundational layer both it and `rover-eval` build on; `screeps-rover-eval` follows the `-eval` convention. No back-compat aliases or re-export shims are kept.

**Deferred out of M0 (later slices, noted so the ADR isn't read as "all done"):**
- the offline **rover driver** (`resolve_moves_via_system` + `SimMoveRequest` / `SimMoveGoal` / `SimMoveCache`, today `screeps-combat-agent/src/pathing.rs:240-518`) ‚Üí `screeps-sim-core` under an optional **`rover` feature** (the crate already declares it), so both the combat squads and the bench import one driver ‚Äî **M1**;
- a kernel movement **recording**: combat-engine's `CombatRecording` is combat-coupled (records damage / deaths / tower intents), so it stayed in the layer; the kernel gets a lightweight movement-trajectory recording when rover-eval needs it;
- the deterministic **`Rng`** (rover-eval reuses combat-eval's for now);
- the **roads** + **loaded-CARRY** physics fixes ‚Äî **M3** (¬ßD3).

**No debt (M0 end state):** git submodule + git-dep + root `[patch]` (no path deps); explicit composition (no `Deref`); direct cross-crate imports (no re-export modules / no constant bridge); no combat concept in the kernel.

#### ¬ßD1b ‚Äî `screeps-rover-eval` (the host-only measurement layer)

`screeps-rover-eval` is a host-only workspace **member** ‚Äî a **library** (`src/lib.rs`) with an `examples/run_register.rs` and an optional `[[bin]]` CLI, the `screeps-combat-eval` shape (register of gated experiments + CPU bench + tuning sweep in one crate) with the corpus-CLI ergonomics of `screeps-foreman-bench`. ### ¬ßD3 The reused server half + the two shared physics fixes

The "server half" ‚Äî the authoritative application of moves with fatigue, contention, and edge-exit ‚Äî is **not new code**. It is `sim_core::resolve_movement` (the movement phases of the combat engine's `resolve_tick`, extracted in ¬ßD1a). The bench feeds it `MoveIntents` carrying **only** `moves` / `pulls`; with no combat actions the combat phases iterate empty maps and no-op (verified against the engine: combat-accumulate + tower loops read empty collections ‚Üí zero effect). The per-tick loop (Tier B):

1. Build `[SimMoveRequest]` from each creep's goal ‚Äî `SimMoveGoal::{To{target,range}, Flee{threats,range}}` with a `MovementPriority`, a `shove` toggle, and an optional `(center,range)` anchor (the real driver request type; `move_to` / `with_priority` / `with_shove` / `with_anchor` builders).
2. `resolve_moves_via_system(&world, owner, &reqs, &mut cache)` ‚Üí per-creep `Direction`s. **This runs the real rover `MovementSystem` + `LocalPathfinder` + resolver ‚Äî the thing under test** ‚Äî and the persisted `cache` carries path reuse + stuck escalation across ticks.
3. `sim_core::resolve_movement(&mut world, &MoveIntents::from(dirs))` **applies** them with engine fidelity (REUSED, **not re-ported**):
   - **Tile contention** by the engine order ‚Äî `rate1` (movers into the vacated tile; forced 100 for a swap), then pulled, then pulling, then moves/weight; heavier loses ties. Stationary creeps are **hard obstacles** (no engine pushing) ‚Äî distinct from rover's *own* friendly-creep shove/swap, which already ran in step 2; the two resolvers are deliberately different and the bench measures the engine's adjudication of what rover requested. A vacating creep lets a follower in same-tick (trains); a failed leader cascades failure down the column.
   - **Fatigue accrual** on an executed step = `(otherParts + loadedCarry) √ó terrain_rate`, `otherParts` = #non-MOVE non-CARRY parts, `terrain_rate` = `1` road / `2` plain / `10` swamp (roads via **SHARED-FIX-1**, loaded-CARRY via **SHARED-FIX-2**, below).
   - **Fatigue regen** every tick = `2 √ó Œ£(MOVE_boost_mult)`, `saturating_sub`, regardless of moving.
   - **Edge zeroing** ‚Äî stepping onto a room-edge tile sets accrued fatigue to 0.
   - **Pull bypass** ‚Äî a pulled creep moves even with nonzero fatigue / no MOVE. (The driver's `pull` / `move_pulled_by` are **no-ops today**, ex-`pathing.rs:337-342`, so the E-family pull-train scenarios require activating pull in the shared driver ‚Äî a SHARED follow-up noted in ¬ßD6/E that benefits combat too.)
   - **Power creeps** ‚Äî no fatigue, 1 tile/tick on any terrain.
   - **Edge relocation** ‚Äî a creep that steps onto a room-edge tile crosses **same-tick** to the neighbour room: the **perpendicular** coordinate flips (`0‚Üî49`), the **parallel** (along-edge) coordinate is **preserved** (engine `creeps/tick.js:52-78`; a creep at `(0,25)` exiting west becomes `(49,25)`, *not* `(49,24)` ‚Äî NOT an `x‚Üí49‚àíx` reflection). This is the **one** behavior a single-room bench inherits unconditionally: keep movers off `x/y ‚àà {0,49}`, or register the owner in `world.npc_owners` (already `pub`) to suppress it. NPC creeps are exempt.
4. Record the frame via the reused `sim_core::record_tick` ‚Üí `SimRecording` (positions + fatigue + issued `Direction` + reason); attach the tick's resolver-event counts + `MovementSystem::tick_stats()` ops for metrics + replay.
5. Advance tick.

> **The two shared physics fixes (both land in `screeps-sim-core`; both default-inert; both improve combat fidelity too).** The engine port omits two fatigue terms that are harmless to combat but fatal to a hauler benchmark; post-extraction they live in sim-core's `terrain.rs` / `body.rs` and benefit **both** sims.
>
> **SHARED-FIX-1 ‚Äî Roads.** Today `fatigue_rate` (ex-`state.rs:22`) returns only swamp-or-plain, and `FATIGUE_RATE_ROAD` (`constants.rs:44`) is a **dead constant** (defined, never referenced ‚Äî grep-confirmed). Add `roads: HashSet<(u8,u8)>` to `SimTerrain` and check it **first** in `fatigue_rate` (road `1` overrides swamp). `resolve_movement` already routes fatigue through `fatigue_rate` ‚Üí **no tick change**; extend `ScenarioBuilder` + the cost sources to place roads and wire rover's already-present-but-empty `roads` `LinearCostMatrix` so the pathfinder *prefers* them. Roads are the single biggest base-ops fatigue lever; without this every fatigue/time-moving metric over a built base is fiction.
> **AS BUILT (M3 ‚Äî 2026-07-01):** `roads: HashSet<(u8,u8)>` on `SimTerrain`; `fatigue_rate` = road 1 ‚Üí swamp 10 ‚Üí plain 2 (engine `movement.js:204-213` order, road-wins-over-swamp). `FATIGUE_RATE_ROAD` is now live. `is_wall` documents the one deliberate divergence (engine `checkObstacleAtXY` treats `wall && !road` as passable, but a road can't be built on a natural wall, so it's unreachable ‚Äî walls stay unconditionally blocking). rover-eval's `cost.rs::fatigue_rate` **delegates to the kernel `SimTerrain::fatigue_rate`** (removed the duplicated swamp/plain branch), and its `TerrainCostSource` populates rover's `roads` sub-matrix (priced at `CostMatrixOptions::road_cost` = 1) *and excludes road tiles from the swamp overlay* so the apply-order (roads then `other.apply_to`) leaves road-on-swamp at 1. Tests: `terrain::road_overrides_swamp_and_plain`, `tick::road_step_accrues_less_fatigue_than_plain`, rover-eval `metrics::road_corridor_across_a_swamp_is_taken`.
>
> **SHARED-FIX-2 ‚Äî Loaded CARRY.** `fatigue_weight` (ex-`body.rs:155`) counts only non-MOVE/non-CARRY alive parts (`body.rs:158`) ‚Äî CARRY is **always free**, with no notion of carried resource. Engine truth: an *empty* CARRY is free, a *loaded* one counts. Add `loaded_carry_parts: u32` to `SimCreep` (default 0) and have `fatigue_weight` also count `min(loaded_carry_parts, alive CARRY parts)`. Because `fatigue_weight().max(1)` is also the `rate4` contention denominator (`movement.rs:156/174`), this *also* corrects loaded-hauler contention ‚Äî a benefit. Loaded CARRY is the *dominant* fatigue term for haulers, the primary subject of this benchmark.
> **AS BUILT (M3 ‚Äî 2026-07-01):** chose `carry_used: u32` (total resources aboard = engine `_.sum(creep.store)`) on `SimCreep`, *not* a pre-counted `loaded_carry_parts` ‚Äî it is strictly more faithful (models the store, and the boost-aware part accounting is derived, not assumed). `SimBody::carry_weight(carry_used)` ports the engine's `calcResourcesWeight` (`movement.js:41`): walk the body back-to-front, each ALIVE CARRY part absorbs `CARRY_CAPACITY √ó BoostTier::carry_capacity_mult` (√ó1/2/3/4) of the load and adds 1 weight until the load is exhausted (naturally capped at the alive-CARRY count). The full weight lives on **`SimCreep::fatigue_weight()` = `body.fatigue_weight()` (structural) + `body.carry_weight(carry_used)`**, and both the mover's rate4 denominator (`movement.rs`) and the tick's accrual (`tick.rs`) now call it ‚Äî so loaded-hauler contention is corrected in the same stroke. Tests: `body::carry_weight_counts_loaded_parts_only` / `carry_weight_is_boost_aware`, `tick::loaded_carry_accrues_fatigue_an_empty_hauler_does_not`.
>
> **Default-inert ‚áí combat byte-identical.** Empty `roads` + `carry_used = 0` make both fixes no-ops for every existing combat scenario (combat creeps carry nothing; combat rooms have no `roads` set), so the combat crates take only a mechanical `carry_used: 0` on their `SimCreep` literals with unchanged behavior ‚Äî the M3 gate is the combat test suites re-passing at their M0 counts (proves inert).

### ¬ßD4 The ground-truth oracle (`oracle.rs`)

Single-creep optima are **exactly computable and already half-shipped** (`gridsearch.rs`); multi-creep optima are **NP-hard** and fall back to relative/regression baselines. The ADR states this asymmetry plainly so no reviewer expects a multi-agent optimum that does not exist.

| Oracle | Method | Cost | Feeds |
|---|---|---|---|
| **Obstacle-aware min step count** | Unweighted 8-connected Chebyshev **BFS** to within `range` of goal | O(2500/room) | `R_len` (¬ßD5.2); the **solvable/unsolvable label** (unsolvable ‚áî goal unreached) |
| **Terrain-weighted min cost** | **Dijkstra** with edge cost `w(dest)` using the *scenario's configured* `plains_cost`/`swamp_cost` (apples-to-apples with what rover's `search` was handed) ‚Äî this is exactly `gridsearch::room_grid_dijkstra` already in the crate | O(2500 log) | `cost_fatigue(O*)` for `R_fatigue` (¬ßD5.1a); `min_ops` lower bound for `ops_efficiency` |
| **Min arrival-time for a body** | **Time-expanded shortest path**: state `(tile, fatigue_bucket)`, edges *move* or *wait* at 1 tick each, edge-tile reset baked in; Dijkstra/Dial over `tiles √ó (max_fatigue+1)`. **Sub-tick ordering pinned to the engine** (`movement.js:11-14` / `creeps/tick.js:105-107`): a move at tick *T* requires `fatigue==0` as evaluated at *T*'s **start**; the destination tile's accrual lands **this** tick; the `‚àí2¬∑Œ£(MOVE_boost)` drain applies once per tick **thereafter** ‚Äî a stall is `fatigue>0 at tick start`, **never** "fatigue would reach 0 after applying this tick's drain" (the latter makes every `R_ticks` optimistic by one tick per stall). | tractable single-creep, 1‚Äì2 rooms | `T*` for `R_ticks` (¬ßD5.1b) and `T_min` (¬ßD5.fatigue) |
| **Closed-form min traversal ticks** | Per-tile stall = `ceil(accrual / drain)` clamped ‚â•1, summed; valid when the body sustains or stalls predictably with no path choices | O(len) | fast-path `T_min`; **cross-checked equal to the time-expanded oracle** on B-family scenarios (guards against a wrong oracle) |
| **Multi-room optimum** | `find_route` for the room sequence, then chain per-room `room_grid_dijkstra` through projected exit tiles | bounded | C-family `R_fatigue` (approximate); gate on single-room exactness + multi-room *monotonic improvement* |
| **Multi-creep makespan** | **NP-hard ‚Äî no exact optimum claimed.** Lower bound = `max over creeps of single-creep T*` (loose). For N‚â§3 on small rooms, a CBS/exhaustive reference gives a true optimal makespan to anchor `flow_efficiency`. **Primary contract = committed regression baseline** (foreman-bench model): gate "no regression beyond X% vs the committed baseline"; absolute improvement is the optimization target the ADR exists to enable. | ‚Äî | D-family quality indicators |
| **Determinism** | No computed optimum ‚Äî the oracle is "the result equals itself across runs/reorders/seeds." | exact | ¬ßD7 fence |

**Oracle ‚Üî executor cross-validation (guards a shared off-by-one):** the closed-form and time-expanded oracles can agree with each other while *both* embed a wrong sub-tick convention. So M2 additionally asserts the oracle's per-tile stall equals the **server half's** observed stall on a single straight corridor ‚Äî validating the oracle against the *engine ordering* (via the executor), not merely against the other oracle.

### ¬ßD5 The metric set (formulas + L-layer mapping)

Two foundational definitions: a tile's **engine fatigue-cost** `w(t)` = `road 1 / plain 2 / swamp 10`; a body's per-step accrual `= (otherParts + loadedCarry) √ó w(t)` with per-tick drain `2 √ó Œ£(MOVE_boost_mult)`, sustaining 1 tile/tick iff `2¬∑Œ£(MOVE_boost) ‚â• (otherParts + loadedCarry)¬∑w(t)`.

**Provenance rule:** wherever possible the metric reads rover's *own* surfaced state (`MovementTickStats { ops_budget_cap, ops_consumed, repaths }`, `MovementResult`, `creep.fatigue()`, `StuckState { ticks_immobile, ticks_no_progress, repath_count, last_distance }`) rather than re-deriving ‚Äî the eval measures the *real* system and owns only the rover metric layer + the oracle (the world and physics are the reused `sim-core`).

#### ¬ßD5.1 Pathing quality (computed on the returned path, before stepping)

- **(a) Fatigue-weighted optimality** `R_fatigue = cost_fatigue(P) / cost_fatigue(O*)`, `cost_fatigue(P) = Œ£_{t‚ààP, t‚âÝs} w(t)`. ‚â•1 always. **Gate (single-creep, A-family): `R_fatigue ‚â§ 1.0 + Œµ`, Œµ‚âà0.02.** [L2 property / L0 on fixed inputs]
- **(b) Tick-to-arrival optimality** `R_ticks = ticks(P, body) / T*(s,g,body)` where `T*` is the time-expanded arrival-time optimum. **The metric that actually matters in-game.** Divergence between (a) and (b) is itself a finding: the cost field is mis-tuned for that body (the single most actionable optimization the bench can surface ‚Äî rover's cost field is **not body-aware** today; a heavy hauler and a 1:1 scout search the same `road=1/plain=2/swamp=10` field). **Gate: `R_ticks ‚â§ 1.0 + Œµ`, regression-tracked per body class.** [L2/L5]
- **(c) Length suboptimality** `R_len = len(P)/len(O*_chebyshev)`; `detour_tiles = len(P) ‚àí len(O*_chebyshev)`. Diagnostic except in uniform terrain. **Gate (uniform-terrain only): `R_len == 1.0`.** [L2]
- **(d) Oscillation/backtrack** `backtrack_steps` / `oscillation` (direction reversals) / `revisit_count`. Targets the operator-observed scatter. **Gate: `oscillation == 0` on static single-creep; `revisit_count` regression-tracked under congestion.** [L2/L5]
- **(e) Incomplete/fail rate** `solvable_fail = fails on oracle-proven-solvable scenarios`. **Gate: `solvable_fail == 0` (failing a genuinely-walled goal is correct).** [L2]
- **(f) Ops vs budget** `ops_consumed` (from `MovementTickStats`), `ops_per_pathfind`, `ops_efficiency = ops_consumed / min_ops`, `budget_pressure = #ticks at cap`. **Gate: single-creep ops ‚â§ base per-room budget (`rooms √ó 2000`, `movementsystem.rs:1310`) on a first, non-escalated search; ‚â§ `MAX_PATHFIND_OPS` (20 000) once stuck-escalation kicks in `ops_multiplier`; curve vs room size & route length.** [L4 invariant + L5 curve]
- **(g) Repath frequency** `repath_rate`, `expiry_repaths` (driven by `reuse_path_length=5`), `stuck_repaths`, `wasted_repaths` (repath returning an identical/worse path). **Gate: `wasted_repaths/repaths` regression-tracked; static-corridor `stuck_repaths == 0`.** [L5]

#### ¬ßD5.2 Movement execution / fatigue (require stepping the server half)

- **Movement efficiency** `movement_eff = ticks_displaced / ticks_trying`, with the wasted ticks partitioned into a disjoint set ‚Äî **the single most important diagnostic**:
  - `idle_fatigued` = ticks `fatigue() > 0` (engine-forced; indicts the **body**, NOT rover) ‚Äî **rover reports these as `Moving` at `:510`, so they are invisible in `MovementResult` and MUST be measured server-side by reading `creep.fatigue()`**;
  - `idle_blocked` = `fatigue==0` but position unchanged AND a contender held the tile (indicts the **resolver/pathfinder** ‚Äî the true congestion-loss class the sim exists to surface);
  - `idle_no_path` = `Failed(PathNotFound)`/`Stuck`; `idle_spawning` = `spawning()` (excluded from `ticks_trying`).
  - **Gate: `idle_blocked / ticks_trying ‚â§ Œµ` in single-creep (should be 0).** [L4 invariant]
- **Fatigue utilization** `fatigue_util = T_min / actual_ticks`; `fatigue_waste = actual ‚àí T_min`; `fatigue_stalls`. Plus the route-independent **over/under-MOVE body diagnosis** `speed_ratio(terrain) = 2¬∑Œ£(MOVE_boost) / ((otherParts+loadedCarry)¬∑w(terrain))` ‚Üí `under_move ‚áî ratio < 1`, `over_move ‚áî min-over-traversed-terrains ratio > 1`. Lets the bench report "this hauler is under-MOVE on swamp (ratio 0.4) ‚Üí 2.5√ó slowdown" and validate the pathfinder routes under-MOVE bodies *around* the terrain that punishes them (links D5.1b ‚Üî this). **Gate: `fatigue_util == 1.0` whenever the oracle says the body is fast enough for the route; `fatigue_stalls` accounted, never blamed on rover.** [L0 closed-form + L5]
- **Congestion** `shove_count`, `swap_count`, `stuck_events`, `train_follow_success`, `deadlock` (N‚â•2 creeps mutually position-stable, each desiring a tile held by another stable creep, ‚â• `report_failure`=12 ticks), `livelock` (Œ£|Œîpos| over a 20-tick window < threshold while `shove+swap` over the same window > threshold), `throughput` (creeps/100t through a chokepoint), `makespan`, `flow_efficiency`. **Ship-blocker gates: `deadlock == 0` and `livelock == 0`; `train_follow_success ‚â• 0.95`; `throughput`/`makespan` regression-tracked.** Targets the operator's "war/squad cohesion / lifecycle hang" failure class. [L4 hard invariants + L5 distributions]
- **Flee/anchor** `flee_min_range_achieved ‚â• 0`, `ticks_to_safety`, `flee_into_corner` (bool), `anchor_violation` (ticks outside `AnchorConstraint.range`), `anchor_work_loss`. **Ship-blocker gates: `flee_min_range_achieved ‚â• 0` whenever escape exists; `anchor_violation == 0` (hard invariant).** [L4]

#### ¬ßD5.3 CPU / algorithmic (op-count primary, wall-clock secondary)

`ops_per_search` (deterministic), `ops_per_resolve` (tile-walkability probes / shove-recursion depth), `ops_per_tick`; `ns_per_search`/`ns_per_resolve`/`ns_per_tick` (regression baseline only, **never a gate** ‚Äî host wall-clock ‚âÝ MMO CPU); `alloc_count` (a counting global allocator in the host bin ‚Äî `LocalPathfinder::run` boxes two `[[_;50];50]` arrays ‚âà 20 KB/call, a flagged pooling opportunity), `peak_path_len`.

**Scaling curves** (the deliverable that catches the "CPU pathfinding death-spiral" failure) ‚Äî sweep one axis, fit & report the exponent, gate "slope ‚â§ baseline + X%":

- `ops_per_tick` vs **N creeps** (1,2,4,‚Ä¶,64) ‚Üí expect ~linear; super-linear = congestion blowup;
- `ops_per_search` vs **route length** (10..600 tiles);
- `ops_per_search` vs **room obstacle %** (open .. maze) ‚Üí A* degradation under walls (worst case = **open room**, which floods to the full `max_ops` with nothing to prune ‚Äî the `bench.rs:84` worst-case construction);
- `resolve_work` vs **congestion density** ‚Üí the `max_shove_depth` recursion is the prime super-linear suspect; instrument its realized depth distribution.

Benched the `bench.rs` way: **`Instant`-timed `#[test]` gates with LOOSE death-spiral bounds (not tight thresholds, not criterion), exact us/op printed** ‚Äî carry forward the `bench.rs` disclaimer that native wall-clock is a *relative* proxy. (Criterion is used nowhere in this repo; the convention is hand-rolled `std::time::Instant` loops.)

#### ¬ßD5.4 The objective function: `w(creep)` ‚Äî the per-movement-intent value weight (designed 2026-07-01, operator-directed)

Everything above scores movement in *physical* units (ticks, fatigue, ops). The **objective** the benchmark optimizes needs one more layer: not all creep-ticks are equally valuable, so the aggregate must weight each movement episode by *what the movement is worth*. The operator's directives: start from **"maximum value transport by distance"** (haulers), generalize to military via **lifetime √ó potential damage/healing**, and best: **the expected value of spending one movement intent** ‚Äî role-dependent but collapsing to a **single scalar**. Designed via a grounded multi-agent synthesis (5 codebase-grounding readers over ADR 0019/0020/0032/0038 + the value kernels, 4 independent formulations, adversarial judging); full transcript retained.

**The scalar.** `w(creep) = quantize(G ¬∑ r_bid)` in **energy-equivalent per tick (e/t)** ‚Äî the marginal energy-equivalent value destroyed if this creep's one movement intent is denied this tick. It is the one-step derivative of `Œ¶ = p_alive ¬∑ r_role ¬∑ W_prod` (expected energy the creep still delivers over its remaining productive window): denying a progress-making move delays the creep's whole remaining production stream one tick, which costs its rate `r` ‚Äî iff that tick actually converts. e/t is the codebase's ONE existing cross-goal currency (stocks: `objective_value.rs::value_e` ADR 0032, `room_net_roi`/`claim_value` ADR 0038; rates: the ubiquitous `body_cost/CREEP_LIFE_TIME` amortization; CPU already priced at 0.02 e/tile) ‚Äî no new unit invented.

**The gates** `G = Œî_crit ¬∑ A ¬∑ S ¬∑ U`, each ‚àà [0,1]: **Œî_crit** ‚àà {0,1} ‚Äî the move reduces fatigue-exact remaining-travel-time `ttr` by ‚â• 1 (0 if fatigued, lateral, off-path, or already in position); **A** ‚Äî arrive-in-time ramp `clamp((ttl ‚àí ttr ‚àí t_min)/t_ramp)` (the operator's *lifetime* intuition: a creep that can no longer arrive alive-and-useful has `w = 0`; ramped, not stepped, to avoid the binary-P(win) cliff); **S** ‚Äî survival veto (net incoming √ó horizon ‚â• hits ‚áí 0; `kite.rs:709` generalized: progress you don't survive to convert is not progress); **U** ‚Äî slack gate (delay costs `r` only while the window binds; slack-rich creeps decay to the amortized-upkeep floor `body_cost/1500/r` ‚Äî a squad with 800 ttl against a 200-tick job bids ~upkeep and correctly loses tiles to a loaded hauler).

**Role table (r_bid, all e/t; conversions only through landed kernels ‚Äî no per-HP energy constant, deliberately):** **hauler** `min(Q/T*_rtt, marginal_route_share)` (Q = 50¬∑CARRY, T*_rtt fatigue-exact oracle round trip ‚Äî THE anchor, `room_net_roi`'s haul term inverted); **military (melee/ranged/heal/dismantle)** ‚Äî one rail: the squad's objective rate `R_O = P(win)¬∑value_e(O)/est_ticks` (the landed force_sizing + objective_value kernels; damage‚Üíenergy and heal‚Üíenergy convert ONLY through the objective they unlock, per the [[combat-ev-economic-and-pwin-gating]] directive), allocated by capability-class spawn-cost shares `Œ±_class¬∑s_member` renormalized ‚â§ 1 per class (fixes the class double-count: a full squad sums to exactly `R_O`, never ~4√ó); for *contention* the squad's **binding member** (critical-path laggard) bids the full `R_O`, others bid the floor ‚Äî the healer a quad waits on outbids a fat hauler (the historical squad-cohesion failure, fixed by construction); **worker/upgrader** `min(WORK¬∑k, supply_rate)¬∑v_sink` (k = 5 build / 1 upgrade e/t/WORK ‚Äî natively energy); **claimer** hard reach gate (`ttr + margin ‚â§ min(ttl, 600)`, `utility.rs:94` verbatim) then hazard-smoothed `min(V, V/max(slack, S_ref))` ‚Äî deadline-tight claimers explode in priority, unreachable ones drop to exactly 0; **scout** `max(Œµ_intel, upkeep)` ‚Äî honestly a policy constant (VOI has no landed kernel; declared, not buried). **In-transit vs in-position:** one function, two derivatives ‚Äî in transit `w` prices Œîttr (a lateral shove is free, matching the observed shove-not-idle resolver); in position the creep bids `G¬∑r` to *hold* its slot but 0 for new travel; tile *quality* within the slot stays ADR 0019 `score_tile`'s job (its clamped [0,1000] preference units are never laundered into e/t ‚Äî ADR 0020 ¬ß2). Escape under fire bids the stock-drawdown *rate* `(V ‚àí recycle_ev)/œÑ_die` ‚Äî same e/t lane; `w = max(w_progress, w_escape)`. **Determinism:** quantize to i64 milli-e/t + stable-creep-id tie-break before ANY ordering (the `quantize_ev` pattern; fence-compatible).

**The aggregate.** Per movement episode: efficiency `Œ∑ = T*/T_realized ‚àà [0,1]` (0 if never arrived / past ttl ‚Äî the catastrophic tail stays visible), sample weight `W = r_bid ¬∑ T*` (the energy-tick value in flight over one optimal episode; squads: cost-shares `Œ±_i` summing each job's `V_O` exactly once). **`H = Œ£ W·µ¢Œ∑·µ¢ / Œ£ W·µ¢`** ‚Äî reported as the weighted mean + seeded-bootstrap 95% CI + p05‚Äìp95 percentiles (`stats.rs::Summary`, byte-reproducible). `1 ‚àí H` ‚âà the fraction of energy-equivalent value destroyed by movement delay. **H is regression-tracked, NEVER a `==1` gate** (per-creep T* under mutual exclusion makes H<1 structural); the ¬ßD5.1/¬ßD5.2 physical gates + the failed-move sentinel (`== 0`) remain the hard pass/fail layer underneath. **Hauler reduction (the operator's starting point, by construction):** all-hauler fleets degenerate to share=1, P(win)=1, U=1 ‚áí `w = œÅ = Q/T*_rtt`, `W = Q`, `Œ∑ = T*_rtt/T_rtt` ‚Äî exactly "maximum value transport by distance" with the distribution + CI, zero residue; the military generalization changes only the `(value, weight)` construction, never the aggregation.

**Open decisions ‚Äî RESOLVED 2026-07-01 (each DECIDED with rationale in `value.rs`'s `## ¬ßD5.4 decisions` block, operator-veto-pending; the military arm is implemented on these defaults; original list kept for the record):** (1) squad bid semantics ‚Äî binding-member-bids-`R_O` for contention vs per-member cost-shares (synthesis recommends binding-member for triage, shares for sample weights; both integrate to `R_O`); (2) slack floor ‚Äî amortized upkeep vs literal 0; (3) the policy constants `Œµ_intel`/`v_sink`/`t_ramp`/`S_ref` (ship defaults + sensitivity sweep proposed); (4) ttl plumbing ‚Äî scenario-level (zero kernel change, proposed v1) vs `ticks_to_live` on `SimCreep`; (5) economic-facts bridge ‚Äî scenario-supplied pre-priced facts (ObjectiveIntel pattern, proposed) vs extracting `room_net_roi`/`claim_value` into a pure crate; (6) non-energy cargo ‚Äî pinned constants (seed-stable, proposed) vs live FairValue; (7) renewal ttl semantics; (8) escape-bid magnitude (`max` vs sum, wounded-creep discount); (9) runtime adoption ‚Äî benchmark-only vs replacing rover's High/Normal/Low resolver priority with quantized `w` (the same kernel would then be *the* triage rule the bot ships); (10) multi-room T* approximation tolerance; (11) H reporting granularity (per-family primary + pooled secondary, proposed).

### ¬ßD7 The determinism fence (first-class ship-blocker)

The invariant: **identical inputs ‚Üí bit-identical outputs across repeated AND reordered runs.** The historical offender was three seed-ordered `std::HashMap` iterations in `resolver.rs` (`topological_sort_follows` consuming the shared ops budget in seed order; `current_pos_to_entity` last-write-wins; swap discovery order), each now fixed with explicit `Handle`-sorted / `(room,x,y)`-sorted tie-breaks (`resolver.rs:146,225,239,297,379,424-443,511-521,574-580`). `MovementSystem` already requires `Handle: Ord`. The eval enforces three sub-gates, mirroring the combat-eval's `sim_is_deterministic_over_rounds` (spread 0):

- `det_repeat` ‚Äî run a scenario R‚â•8 times from the same seed/world ‚Üí all metric vectors bit-identical (spread 0);
- `det_reorder` ‚Äî permute the creep-insertion order P ways ‚Üí identical final trajectories & metrics;
- `det_hash_seed` ‚Äî run under K different process-global HashMap seeds (a seeded `BuildHasher` injected into the harness, or run the binary K times with randomized SipHash keys and diff) ‚Üí identical results.

The hash covers the **full per-tick recording** (positions + fatigue + issued directions + the built cost-matrix bytes ‚Äî the `SparseCostMatrix` HashMap is the latent hazard to include). **Gate: spread == 0 on all three; any nonzero spread is a ship-blocker that localizes a leaked HashMap iteration.** Debugging method (the known one): thread-local capture of each resolver iteration + per-run diff. **`det_reorder` must be cheap to run on *every* scenario** so a leaked iteration is caught the instant it appears, not only in a dedicated test. (`param_sweep.rs` additionally pins `sweep_point_is_deterministic` via `f64::to_bits` run-twice, the combat-eval pattern.)

## Consequences

### Positive

- **The operator's headline asks become numbers** ‚Äî `R_ticks` / `fatigue_util` / `movement_eff` quantify "time creeps spend moving" and "efficient usage of fatigue"; `R_fatigue` / `R_len` / `oscillation` quantify "optimal pathing"; the scaling curves + ops gates quantify "algorithmic/cpu benchmarking."
- **The "validated separately" handoff from ADR 0023 is finally a real artifact**, not a promise. Rover regressions are caught directly and offline, not accidentally via a downstream combat flake.
- **The single most actionable optimization is surfaced and measurable**: rover's cost field is not body-aware, so `R_fatigue==1` can coexist with `R_ticks>1` for low-MOVE bodies. The A4/B3 scenarios turn that into a gate and a target.
- **Determinism becomes a directly-targeted first-class fence** on the documented historical offender, runnable on every scenario ‚Äî the strongest "in-voice" gate for this codebase.
- **Cross-vouching with the live canary**: the offline bench emits the same G-13 wasted-move-rate and ADR-0004 ops-saturation numbers the seg-57 stream watches, so each validates the other (the 0006 pattern).
- **No duplicated mover, and a reusable mechanism foundation.** The bench runs the bot's *real* rover over the combat sim's *reused* world/physics/driver ‚Äî one mover, one server half, validated once. The `screeps-sim-core` extraction gives every future offline sim (economy, hauling, lifecycle ‚Äî ADR 0028) the same substrate, and turns the dead `FATIGUE_RATE_ROAD` constant live for **both** sims.

### Negative / costs

- **The `sim-core` extraction is real, one-time engineering** ‚Äî splitting `resolve_tick` into `sim_core::resolve_movement` + `resolve_combat_tick`, recomposing the world as `SimWorld` + `CombatState`, and the `Combat*`‚Üí`Sim*` rename touch `combat-engine` / `agent` / `eval`. The hard guard: the recomposed combat tick must be **byte-identical** to today's ‚Äî the existing combat determinism fence (spread 0) + the FP/FN calibration are the gate (¬ßM0). *After* it, the server half is **reused, not re-ported**, so there is no second physics half to drift from the engine.
- **Overfit-to-sim risk** ‚Äî the bot could be tuned to ace the sim and regress live. Mitigated by the 0006 fidelity triple (conformance golden vectors captured from the live server / parity report / MMO canary alignment) and by holding mechanism fixed while sweeping *policy* (the sweep), not the reverse.
- **Multi-creep has no exact optimum** ‚Äî D-family gates are invariants (no deadlock) + committed regression baselines, not optimality. The ADR states this asymmetry so reviewers don't expect more.
- **Two new crates + an engine refactor**, not one ‚Äî but `sim-core` is mechanism code factored *out* of an existing crate (net new lines roughly flat), `rover-eval` is plumbing-identical to three existing host crates, and the refactor is one-time. The SHARED-FIXES (roads, loaded-CARRY) are additive and default-inert.

### CPU & tick-safety impact

- **Zero new live CPU; no reset.** `screeps-rover-eval` is host-only (rayon/clap), excluded from all three wasm builds. The `sim-core` extraction is a host+wasm **refactor** with no serialized-shape change ‚Üí **no `WORLD_FORMAT_VERSION` bump, no reset** (the bot's saved state is untouched; `combat-engine`/`sim-core` are sims, not bot state). The `rover` feature only adds the wasm-safe `screeps-rover` dep, already in the bot. This mirrors 0006:132's "the harness adds no MMO cost."
- The new `pub` surface (`SimWorldCostSource::from_world`, the moved `Rng`) is purely additive; the two SHARED-FIXES are **default-inert** (every existing combat scenario stays byte-identical, proven by the fence).
