# ADR 0009 — Room Planning & Multi-Room Layout

- **Status:** Proposed
- **Date:** 2026-06-09
- **Deciders:** William Archbell
- **Related:** IBEX-036 (unbudgeted gather BFS + uncached `describe_exits`), IBEX-037 (seg-60 planner restart-thrash on fingerprint mismatch), IBEX-032 (claim/colony/scout gate home rooms by *linear* not route distance); Field Report C (CPU death-spiral — planning is a burst consumer); review report §2-#8 (Room data, visibility & planning — "one of the few real load-shedding mechanisms (room-plan CPU budget)") and §8 Sequencing. Builds on / cross-refs [0004](0004-cpu-governance-and-load-shedding.md) (global `CpuGovernor` + budgeted pathfinding facade; IBEX-036 shed tier), [0006](0006-eval-and-iteration-harness.md) (offline harness + colony-health score + seg-57 metrics), [0002](0002-serialization.md) (seg-60 planner state lives on the same brittle bincode substrate; segment-map discipline), [0005](0005-runtime-and-scheduling-model.md) (tick-level panic containment around the planner system; resumable work is the one place specs already does multi-tick), [0001](0001-entity-model.md) (multi-room graph keyed by stable `RoomName`, never recyclable Entity indices).

## Context

The room planner is `screeps-foreman` (workspace member, patched local path), driven from `room/roomplansystem.rs`. It is **already among the strongest subsystems in the codebase** (review §2-#8, §6 maturity 4/3/2/3/3) and does *not* need a rewrite — it needs (a) two confirmed reliability fixes, (b) targeted layout-quality tuning, and (c) a genuinely new capability the bot lacks: **multi-room layout** (inter-room road planning to remote sources/exits, and a room-graph the per-room planner and remote-mining/expansion can both consume). The operator has found the offline bench (`screeps-foreman-bench`, the workspace `exclude`) useful and wants offline experimentation tied into the eval harness.

Forces and constraints:
- **Single-threaded WASM.** No threads/locks; the only concurrency model available is the planner's existing cooperative *multi-tick* resume (a deliberate, narrow use of resumable work per [0005](0005-runtime-and-scheduling-model.md)). The bench may use rayon — but **only offline on the host target**, never in the bot.
- **CPU = execution + intents.** Planning is *pure execution* (no intents until `ConstructionMission` emits `create_construction_site` / `destroy` — `missions/construction.rs:249-262`). The planner is a **burst** consumer: distance transforms / flood-fills / Dinic's min-cut / A* roads are expensive but amortizable across ticks against the bucket. Inter-room road planning adds `find_route`/cross-room search, which is exactly the kind of un-budgeted pathfinding [0004](0004-cpu-governance-and-load-shedding.md) governs.
- **VM-reset resilience.** Planner progress is persisted to seg-60 (`PLANNER_MEMORY_SEGMENT = 60`, `roomplansystem.rs:241`) and a completed `Plan` is a serialized `RoomPlanData` component. A reset mid-plan must resume, not silently restart forever (IBEX-037).
- **Finite creep lifetime is upstream of layout, not in it.** The planner produces a static layout + RCL-ordered build sequence; replacement/pre-spawn/retirement of the *creeps* that build and use it are the spawn/mission layers' concern. The one place lifetime touches layout is **reachability** (a spawn whose only open adjacent tile is a dead-end pocket traps every newly-spawned creep — already handled, `reachability.rs:179-218`).
- **Incremental & confidence-driven.** The per-room planner is a stable seam (`PlannerRoomDataSource` in, `Plan` out, `RoomVisualizer` for rendering); changes hide behind it. Multi-room is *additive* — a new layer of planning above the per-room planner, not a change to it.

### What the planner is today (assessment)

**Architecture — a lazy depth-first search tree over placement layers** (`search.rs`). `SearchEngine` drives a DFS over an ordered `Vec<Box<dyn PlacementLayer>>` (the default 23-layer stack, `layers/mod.rs:75-101`). Each layer implements `PlacementLayer::candidate(index, state, analysis, terrain) -> Option<Result<PlacementState,()>>`: `None` = exhausted (pop frame), `Err(())` = structurally invalid (skip), `Ok(state)` = a candidate placement. Candidates are generated **one at a time** so memory is proportional to tree *depth*, not width (`search.rs:18-26`). The pipeline has three phases (`pipeline/mod.rs:47-65`): `Analysis` (distance transforms / flood-fills) → `Searching` (the tree) → `Finalizing` (build order + rich `Plan`), with `Complete`/`Failed` terminals.

**Strong points:**
- **Genuinely multi-tick & CPU-budgeted.** `CpuBudget` is a `Box<dyn Fn() -> bool>` polled inside every loop (`pipeline/mod.rs:11-34`); the analysis phase checks it between *every* flood-fill (`analysis.rs:110,146,161,…`) and the search checks it each candidate (`search.rs:183,289`). `roomplansystem.rs:188-191` builds the budget from remaining tick CPU at **0.75 of allocated** so it stops early and never overshoots the tick — this is the "one of the few real load-shedding mechanisms" the review credits (§2-#8). It also gates on `bucket >= bucket_threshold` and reserves 20% headroom (`get_cpu_budget`, `roomplansystem.rs:260-295`).
- **Sound, library-grade layout algorithms.** Two-pass Chebyshev **distance transform** (`terrain.rs:258-311`); BFS **flood-fill** from sources/controller/mineral/exits stored compactly as `u16` (`terrain.rs:160-207`, halving seg size); a **true max-flow / min-cut** rampart planner using vertex-splitting + Dinic's (`defense.rs:282-431`) with checkerboard wall/rampart classification (`defense.rs:159-206`) and an exit-flood validation that rejects undefended plans (`defense.rs:443-544`); **A\*** road routing that reuses existing roads at cost 0 to grow a shared trunk, penalizes swamp 6× and containers, and rejects routing through buildings (`road_network.rs:335-440`); a **reachability** BFS that rejects unreachable structures, excessive detours, and dead-end spawn pockets (`reachability.rs`).
- **Clean offline/online split.** The whole crate compiles on native targets; game-API calls (`execute_operations`, `snapshot_structures`, `impl RoomVisualizer`) are gated behind the `screeps` feature (`Cargo.toml`, README). `Plan` is *pure data* and execution is policy-injected via `ExecutionFilter`/`CleanupFilter` traits, implemented game-side in `missions/construction.rs`. This is what makes the bench possible.
- **Search hygiene.** Score-margin pruning (`prune_margin`, `search.rs:251-257`), a **layer fingerprint** (hash of layer names, `search.rs:332-338`) that validates a resumed engine against the current layer config, a generic `StampLayer` (`stamp_layer.rs`) reused for both hub and lab stamps, six dedicated scoring layers (`anchor_score`, `hub_quality_score`, `extension_score`, `tower_coverage_score`, `upgrade_area_score`, `upkeep_score`) feeding a weighted `PlanScore`, and an `RclAssignmentLayer` post-process that orders structures by hub flood-fill distance (`rcl_assignment.rs`).

**Weak points (confirmed):**
- **IBEX-037 — restart-thrash with no convergence bound.** On a fingerprint mismatch, `roomplansystem.rs:197-204` restarts from `PlannerBuilder::default().build()` every tick with only an `info!` log. If the layer config is *permanently* changed (a deploy that reorders/renames layers — note `compute_fingerprint` hashes layer *names*), `resume` fails forever, the accumulated `PlanningState` is discarded each tick, and the room burns its whole planning budget producing nothing → no layout → no construction → stunted growth. No restart counter, no backoff. On encode failure (`roomplansystem.rs:443`) the segment is silently left un-updated.
- **IBEX-036 — unbudgeted candidate-room BFS.** `room/gather.rs:118-229` expands every reachable room to `max_distance` calling `game::map::describe_exits` *live* per room (`gather.rs:143,181`) instead of the already-cached `RoomData.exits`, with no `game::cpu` read or early-out. Bounded (gated behind a ~500-tick discover interval; frontier only expands through already-scouted non-hostile rooms) but a real avoidable burst. This BFS is *also the natural place a room-graph should live* — see the multi-room design.
- **Route-blindness at the empire layer (IBEX-032).** Expansion eligibility (`claim.rs`, `colony.rs`, `scout.rs`) gates home rooms by **Manhattan** room-delta, not route distance — it can pick layouts/claims that are linearly close but route-unreachable. The per-room planner is excellent; the *placement of rooms relative to each other* and the roads between them is unplanned.
- **No multi-room layout at all.** `Plan.road_network` (`plan.rs:267`) is intra-room only. The planner has zero concept of *which exit* a remote source sits behind, or of a shared trunk road that should leave the base toward a cluster of remotes. Remote-mining roads are grown ad-hoc by miners/haulers, not planned, so they are neither optimal nor coordinated, and the hub anchor is chosen with no awareness of where the room's productive neighbors are.
- **Minor:** anchor `AnchorLayer` rebuilds + sorts the full candidate list on *every* `candidate(index,…)` call (`anchor.rs:62-82`) — O(tiles·log) repeated per index (acceptable for one room, wasteful at scale); scoring weights are hand-tuned constants never validated against outcomes (the bench can close this — see §4).

## Decision

Keep the per-room search-tree planner as the stable seam. Make three decisions:

### D1 — Harden the existing per-room planner (closes IBEX-037; reliability, None/Memory-format)

1. **Bounded restart + backoff (IBEX-037).** Add a `restart_attempts: u32` (`#[serde(default)]`) to `RoomPlannerRunningData`. On a `resume` `Err` (fingerprint mismatch), increment and restart **only** while under a small cap (e.g. 3); past the cap, set `RoomPlanState::Failed { time: game::time() }` and back off via the **existing 2000-tick replan gate** (`roomplansystem.rs:348`) so the room stops burning its budget producing nothing. Log encode failures (`:443`) at `warn`, not silent skip.
2. **Fingerprint stability note.** `compute_fingerprint` hashes layer *names* (`search.rs:332`), so reordering or renaming layers between deploys invalidates every in-flight plan. Document this as a deliberate **layer-config = breaking-for-in-flight-plans** contract; completed `Plan`s are unaffected (they are `RoomPlanData`, not `PlanningState`). When the multi-room layers below are added to the default stack, that is a one-time in-flight-plan reset, not a `Plan`-format break.
3. **Cheap anchor fix.** Memoize the sorted anchor candidate list across `candidate` calls within one frame (or precompute it in the analysis phase as a sorted `Vec<Location>`), removing the repeated O(tiles·log) rebuild (`anchor.rs:62-82`).

### D2 — Layout-optimization improvements, grounded in prior art (quality; Behavioral / in-flight-plan reset)

Foreman already does the *hard* parts of the prior-art canon. The table makes "have vs add" explicit; only the **add** rows are new work.

| Technique (prior art) | Foreman status | Action |
|---|---|---|
| **Distance transform** (open-space scoring) | **Have** — `terrain.rs:258-311`, used by `AnchorLayer` (`min_dist_transform=3`) and `anchor_score` openness/defensibility. | Keep. Expose DT-≥-N gating as a tunable for larger stamps. |
| **Stamp / bunker packing** | **Have** — generic `StampLayer` + hub/lab stamps (`stamp_layer.rs`, `stamps/`). | **Add:** make stamp *sets* swappable via the builder so the bench can A/B a bunker stamp vs. the current fast-filler hub. |
| **Min-cut / max-flow ramparts** | **Have** — true Dinic's vertex-split min-cut (`defense.rs`), the most advanced item in the canon. | Keep. Optionally **weight** the cut toward terrain chokepoints (clarkok-style weighted edges) — bench-gated. |
| **Hub/anchor placement scoring** | **Have** — DT openness + source/controller/mineral distance + exit proximity + a defensibility proxy (`anchor_score.rs`). | **Tune** weights against bench outcomes (§4); **add** a multi-room term (see D3). |
| **Flood-fill tile ranking** | **Have** — hub flood-fill drives RCL ordering (`rcl_assignment.rs:53-77`) and extension scoring. | Keep. |
| **Tower coverage scoring** | **Have** — `tower.rs` + `tower_coverage_score.rs`. | Tune weight. |
| **Road-network quality / trunk merging** | **Have** — A* with road-reuse, swamp/container penalties, adjacency tie-break (`road_network.rs`), plus a `RoadPruneLayer`. | **Add:** the inter-room trunk extension in D3. |
| **Scoring-weight tuning** | **Partial** — six scoring layers exist; weights are hand-set constants, never validated. | **Add:** bench-driven weight calibration tied to the colony-health score ([0006](0006-eval-and-iteration-harness.md)). |
| **Upgrade-area work-slot optimization** (operator-directed, 2026-06-10 — born from the live range-mismatch bug) | **Partial** — `controller_infra` places container/link; `upgrade_area_score` scores the area; but placement does not maximize **work slots** (walkable tiles ≤1 from the container AND ≤3 from the controller — park-and-work without moving; matters most early with many upgraders, less with one big one). | **Add:** `controller_infra` picks the container tile maximizing the work-slot count; the link goes adjacent on the tile costing the FEWEST slots (it's unwalkable); slot tiles avoid the planned road network (penalize, don't forbid — upgraders parked on corridors block traffic); the existing reachability validation must still pass and the bench corpus (incl. tightly-constrained rooms) must show no plan regressions — the operator's congestion/pathing guard. Export the container↔controller **range contract as a foreman constant** the bot reads (the 2-vs-3 mismatch class becomes unrepresentable). `upgrade_area_score` rewards slot count so the search prefers good upgrade areas. |
| **CPU burst amortization (plan-once, cache-minimal)** | **Have** — multi-tick `CpuBudget`, seg-60 resume, completed `Plan` cached as a component; execution diffs plan vs. live every 50 ticks (`construction.rs:245`). | Keep; this is the glitchassassin "Architects" pattern already realized. |

**The single most valuable D2 add is scoring-weight calibration**, because foreman's algorithms are sound but its *objective* (the weights) has never been validated against whether the resulting base actually grows/survives faster. The bench + harness make this measurable rather than guessed.

### D3 — Multi-room layout (the genuinely new capability)

Introduce a **room-graph / connectivity model** keyed by stable `RoomName` (never Entity indices — [0001](0001-entity-model.md)) and an **inter-room road planner** that the per-room planner, remote-mining, and expansion all consume.

**1. Room-graph model (`RoomGraph`).** A persisted adjacency structure: nodes are rooms (owned, reserved-remote, scout-known, unknown), edges are exit transitions (the 4 `describe_exits` directions) annotated with **route distance** (from `RoomRouteCache`, budgeted via [0004](0004-cpu-governance-and-load-shedding.md)) and a traversability flag (reuse `can_traverse_between_room_status`, `gather.rs:150`). This is built by **generalizing the existing gather BFS** (`gather.rs:118-229`) — that BFS *already* walks the room graph; it just discards the topology after picking candidates. Fixing IBEX-036 (cache `RoomData.exits` instead of live `describe_exits`; add a `CpuBudget` closure that persists a partial frontier; cap visited count) and **persisting the resulting graph** are the same piece of work. The graph is keyed by `RoomName`, so it is reset-resilient and never dangles.

**2. Per-room "exit affinity" feeds the anchor.** When a room has known productive neighbors (remote sources in adjacent rooms via the room-graph), add an optional **`AnchorScoreLayer` term** that biases the hub toward the exit(s) leading to those neighbors — so the base's trunk road naturally faces its remotes. This is the only change to the per-room planner, and it is additive (the layer scores 0 weight when no neighbor data is supplied), so a single-room plan is unaffected. The neighbor data is passed in via an **extension to `PlannerRoomDataSource`** (a new `get_neighbor_affinity()` defaulting to empty) so the seam stays clean and the bench can supply synthetic affinities.

**3. Inter-room road planner (`InterRoomRoadLayer`, run *above* the per-room planner).** A separate, multi-tick planning pass that, for each owned room and each chosen remote (source-keeper-aware: SK rooms = `(x%10)∈{4,5,6} && (y%10)∈{4,5,6}` minus center; remotes scored by route distance per the remote-mining economics — efficiency falls off sharply past ~100 tiles), routes a road **base-hub → exit tile → remote container** by composing the per-room A* (`road_network::find_path`, reused per room) across the room boundary: route to the exit in the home room, cross the edge, route from the entry tile to the remote container in the neighbor. Roads are stored per-room so each room's `Plan` (or a sibling `RemoteInfraPlan`) owns its own tiles; the trunk merges onto the in-room road network at cost 0 exactly as the intra-room planner already does. Output integrates with **remote-mining missions** (`miningoutpost`) as a planned road set the mission builds/maintains, instead of ad-hoc per-creep road growth.

**4. CPU-bounded, multi-tick, reset-resilient — by construction.** The inter-room planner reuses the *exact* mechanisms that already make the per-room planner safe:
- It is a **`PlanningState`-style multi-tick pass** polling the same `CpuBudget`, sharing the seg-60 planner slot (one room/remote per budget slice), so it inherits the 0.75-of-budget early-stop and the `bucket_threshold` gate.
- All cross-room search (`find_route`, the room-boundary A*) routes through the **budgeted pathfinding facade** of [0004](0004-cpu-governance-and-load-shedding.md) and is **shed first under Conserve/Critical** alongside discovery bursts and expansion (the IBEX-036 shed tier). A starved colony does not plan remotes; it keeps its existing base and its `MIN_PATHFIND_OPS` floor.
- The `RoomGraph` and inter-room road sets are **persisted** (`RoomName`-keyed) and warm on reset, so the post-reset tick does not re-run the room-graph storm — consistent with [0004](0004-cpu-governance-and-load-shedding.md)'s persist/warm direction and the [0002](0002-serialization.md) seg discipline (the planner already owns seg-60; the graph is a labelled addition there or a dedicated free segment — **not** 50-55/56/57/60/99).

**5. Replaces linear distance at the empire layer (IBEX-032).** Expansion eligibility (`claim.rs`, `colony.rs`, `scout.rs`) reads **route distance from the room-graph** instead of Manhattan delta, with a reachability check, so claimers/escorts are never sent to linearly-close-but-route-unreachable rooms.

> **Scoring substrate landed early (P0.P7, 2026-06-10):** `screeps_foreman::room_scoring` now exists — the shared composable candidate-scoring module (RoomScorer trait + WeightedPipeline + ScoringContext; prospector's stage-1 is already a consumer with bit-identical rankings). The Inc-7 form of this item: `claim.rs` replaces its bespoke `source_score`/`walkability_score`/`distance_score` with the `WeightedPipeline` — core scorers over `RoomFacts` built from live `RoomData` terrain/statics, composed via `ScoringContext` with `RouteDistanceToOwnedScorer` fed from `RoomRouteCache` route distances (absent entry = unreachable, gated before scoring) and `ProximityToSelectedScorer(Near)` over owned rooms — keeping foreman `PlanScore.total` as the separate stage-2 term it already uses.

## Alternatives Considered

| Option | Pros | Cons |
|---|---|---|
| **Harden + tune the existing planner; add multi-room as an additive layer above the stable per-room seam** (chosen) | Preserves the strongest subsystem; multi-room reuses proven multi-tick/budget/A* machinery; bench + harness validate quality; reset-resilient by `RoomName` keying | Touches gather BFS + expansion gating; new persisted graph adds (bounded) seg pressure |
| Rewrite the planner (e.g. to a fixed bunker stamp + cheaper heuristics) | Simpler, cheaper per plan | Discards a library-grade min-cut/search-tree planner the review rates highly; no evidence the current cost is a problem (it is budgeted); throws away the offline bench investment |
| Plan multi-room inside the per-room search tree (one giant multi-room search) | One pipeline | Combinatorial blow-up; breaks the clean single-room `PlannerRoomDataSource` seam and the bench's per-room model; CPU-unbounded across rooms |
| Keep remote roads ad-hoc (status quo), only fix IBEX-036/037 | Smallest change | Leaves the bot route-blind and remote roads uncoordinated — the operator's stated gap; expansion stays linear-distance (IBEX-032) |
| Use full cross-room `PathFinder.search` (multi-room) for inter-room roads | Optimal paths | Far more expensive than `findRoute`-then-per-room-A*; un-amortizable; exactly the un-budgeted search [0004](0004-cpu-governance-and-load-shedding.md) governs |

## Consequences

**Positive**
- IBEX-037 closed: a room can no longer thrash its planning budget forever; it fails-and-backs-off deterministically.
- IBEX-036 closed *and repurposed*: the gather BFS becomes budgeted and its topology is retained as the persisted `RoomGraph` instead of discarded.
- The bot gains planned, coordinated **base↔remote road networks** and a route-aware empire layer (IBEX-032), removing the route-blindness the competitive verdict (review §Exec) calls out for expansion.
- Layout *quality* becomes measurable and tunable via the bench + colony-health score, rather than hand-guessed weights.
- Offline experimentation (the operator's "foreman has been useful") is formalized and tied to the eval harness — a fast iteration loop that never touches the live tick.

**Negative / new risks**
- **Multi-room is a new CPU consumer.** Mitigated by riding the *same* `CpuBudget` + facade + shed-tier as the per-room planner; it is sheddable, multi-tick, and persisted-warm. The harness CPU-pressure scenario ([0006](0006-eval-and-iteration-harness.md)) must assert that enabling multi-room planning does **not** regress CPU headroom or trip the death-spiral alarm.
- **New persisted state (`RoomGraph` + inter-room road sets).** Adds bounded seg pressure on the same brittle bincode substrate as seg-60; must respect [0002](0002-serialization.md)'s segment-map discipline and `#[serde(default)]` for any added field, and must reject-and-reset (not silently empty) on decode failure once [0002](0002-serialization.md) Stage 1 lands. Label its introduction explicitly.
- **Scoring-weight changes shift which layout wins** — a Behavioral change that also invalidates in-flight `PlanningState` (fingerprint), causing a one-time replan of rooms mid-plan. Confined to low-stakes ticks; completed plans persist.

**CPU / tick-safety.** All new work is execution (no new intents beyond the existing construction-site/destroy emission). It is budgeted, sheddable, and multi-tick. The whole planner already sits inside the per-system run; with [0005](0005-runtime-and-scheduling-model.md)'s tick-level `catch_unwind` a planner panic can no longer skip `serialize_world`. Bench/weight-calibration runs are **offline on the host target** and never charge the tick.

## Incremental Migration Path

The stable seams: `PlannerRoomDataSource` → `Plan` (per-room, unchanged) and a **new `RoomGraph` resource** above it. Each step is harness-validated ([0006](0006-eval-and-iteration-harness.md)) before the next; never break the running bot mid-increment.

1. **Planner reliability (Increment placement: alongside Increment 1, with the governor — None/Memory-format).** Add the bounded `restart_attempts` + Failed-backoff (IBEX-037), `warn` on encode failure, and the cheap anchor memoization (D1). Budget the gather BFS and source exits from cached `RoomData.exits` (IBEX-036) — this is also the [0004](0004-cpu-governance-and-load-shedding.md) Conserve-tier shed work. **Validate:** force a permanent fingerprint mismatch in sim → assert the room stops replanning after the cap and backs off; profile the discover burst → assert it is budgeted. **Breaking:** None (Memory/format only for the `serde(default)` `restart_attempts` field).
2. **Offline weight calibration + bench↔harness tie-in (anytime after Increment 0 — None-breaking, offline only).** Drive `screeps-foreman-bench` (`plan`/`compare`/`list-rooms`, map JSON in `resources/`, rayon-parallel, PNG + plan-JSON + score-table out) from the harness as a **pre-deploy plan-quality gate**: regenerate plans for a fixed room corpus, assert no score regression and no `Failed`/unreachable plans, and feed the per-room `PlanScore` into the colony-health "economic growth" term ([0006](0006-eval-and-iteration-harness.md) §9). Use it to **calibrate the scoring-layer weights** (D2) against actual bringup outcomes. **Validate:** bench corpus runs green; weight change improves (or holds) the colony-health score on the economy-bringup scenario. **Breaking:** None (bench is host-only).
3. **Room-graph model (after Increment 0 telemetry + the budgeted facade of Increment 1 — Memory/format, labelled).** Generalize the budgeted gather BFS into a persisted `RoomName`-keyed `RoomGraph` (route-distance edges from the budgeted `RoomRouteCache`); swap expansion eligibility (`claim`/`colony`/`scout`) from Manhattan delta to graph route distance + reachability (IBEX-032). **Validate:** place a hostile/closed room between home and a linearly-near candidate → assert it is rejected; force a reset → assert the graph reloads non-empty and the post-reset tick does not re-run the room-graph storm. **Breaking:** Memory/format (new persisted graph; labelled, `serde(default)`, reject-and-reset once [0002](0002-serialization.md) Stage 1 lands).
4. **Inter-room road planner + per-room exit-affinity (after the room-graph + budgeted facade — Behavioral + in-flight-plan reset).** Add the optional exit-affinity `PlannerRoomDataSource` term (D3.2) and the `InterRoomRoadLayer`/`RemoteInfraPlan` multi-tick pass (D3.3), sharing seg-60's budget slice and routed through the facade; wire its output into `miningoutpost` as planned roads. **Validate:** a remote with a planned trunk → assert the road set is generated within budget, merges onto the in-room network, and the mission builds it; under induced CPU pressure assert the inter-room pass is shed first and base operation is unaffected (no headroom regression vs. baseline). **Breaking:** Behavioral (remote roads now planned, not ad-hoc) + a one-time in-flight-plan reset when the exit-affinity term changes the layer fingerprint (completed `Plan`s persist).

**Breaking-change labels:** Step 1 — None (Memory/format for one `serde(default)` field). Step 2 — None (offline). Step 3 — Memory/format (labelled, new persisted graph). Step 4 — Behavioral + a one-time in-flight-plan replan (completed plans unaffected). No `Plan`-format break and no whole-state drop is introduced by this pillar; all additions are `serde(default)` and the only resets are the deliberate, labelled in-flight `PlanningState` replans on layer-config change.
